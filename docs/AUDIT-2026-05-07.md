## Executive summary

The rev3 audit finds the rev2 fix waves were largely successful — most findings cleanly closed with regression tests. But the adversarial pressure surfaced three real issues the per-team scans missed, including one BLOCKER and two HIGHs that were *introduced by the fixes themselves*.

**1 BLOCKER candidate** (vercel-ai adapter):
- **R3 — `runner.ts:181-193` `maxSteps` never threaded to `streamText`.** The AI SDK's default `stopWhen: stepCountIs(1)` kicks in. The runner terminates after **ONE** LLM step regardless of agent loop intent. All 4 runner tests use a `_streamText` stub that fakes multi-step behavior, so the test surface didn't catch it. **The reference adapter shipped in W4 to prove the "any framework runs in a pod" claim does not actually run a multi-step loop.**

**2 NEW HIGHs introduced by fixes:**
- **C1 NEW-H1 — `safeRestart` `attempts` counter never reset.** `reset()` exists in `informer-restart.ts:153-158` but is captured and never called from `watch.ts` or `job-watch.ts`. What the JSDoc calls `maxConsecutiveFailures` is actually `maxTotalNonConcurrentErrors`. Twelve independent transient flaps spread over weeks (all recovered) permanently wedge the informer cap. The operator goes dead for AgentTask reconciliation while `/healthz` continues reporting 200 OK for up to 5 minutes.
- **C3 REV3-H1 — B5 fix is partial.** `MIN_SAFE_MIN=1` was applied to model-watch and the PATCH validator, but NOT to `model-index.ts:147` (`ModelIndex.lookup()` returns unclamped `minSafe`). The router calls `aimd.updateBounds` with that unclamped value on every request, overwriting the watch-time clamp. A CR with `spec.minSafe: 0` triggers AIMD cap collapse after the first request. The original B5 DoS is restored.

**Strategic posture (R1 + R2):**
- **Clause (a) — substrate-thin context-pressure handling — SURVIVES, supporting argument WEAKENED.** The rev2 framing "no surveyed competitor ships any context-pressure primitive" was overstated. Anthropic's `compact-2026-01-12` beta is now operator-tunable (default 150K, min 50K). Microsoft Agent Framework 1.0 ships a substrate-thick `CompactionTrigger`/`TokenBudgetComposedStrategy` surface (experimental). The honest framing: every surveyed substrate that ships context-pressure primitives chose substrate-thick auto-compaction; kagent is the only OSS substrate that bets the OPPOSITE way (refuse + introspect + flag).
- **Clause (b) — sealed-JWT capability narrowing — WEAKENS at marketing surface, SURVIVES technically.** Solo.io kagent **enterprise** (commercial) shipped controller-minted RS256 JWTs with JWKS publication. Same wire shape; different primitive (identity-OBO, not capability-narrowing-on-spawn). The OSS K8s-native gap at SIG Apps remains uncontested — but kagent's `WHY.md` §1.b language needs to add "OSS, not enterprise" and "narrowing-on-spawn, not OBO-identity" qualifiers.
- **A2A bridge architecture in `A2A-IMPLEMENTATION-PLAN.md` is wrong** (R2). Bridge approach loses the "speaks A2A natively" qualifier. Recommend Option D (in-pod A2A server, transparent-proxy pattern matching AgentCore + Vertex/ADK).
- **12-month moat claim is too optimistic.** R2 estimates 4–7 months actual moat. OpenAI/Cloudflare can clone the four-piece composition in 2–3 weeks of SDK work.
- **V0.1-COMPARISON.md is ~60% runnable** (R2). Five missing deliverables: daily-check script, agenttask template, topic 4 manifests, pricing reconciliation, dry-run preconditions probe.
- **Recommended slate re-ordering:** ship capability-narrowing RFC (S5) BEFORE A2A wire (S3). The contribution-back path is the only way to convert a 4-7-month moat into permanent positioning.

## R3 — In-pod SDK landscape (rev3 resurvey)

> Full report: `evidence/audit-rev3/R3.md` (237 lines)

**Headline:** the vercel-ai adapter is a **partial Potemkin proof**. The most damaging finding is a single-line defect that was missed in W4 review.

**`packages/agent-loop-vercel-ai/src/runner.ts:181-193`** — `maxSteps` is declared in the input but **never threaded to `streamText`**. The AI SDK's default `stopWhen: stepCountIs(1)` kicks in. The runner terminates after **ONE** LLM step regardless of agent loop intent. All 4 runner tests in `runner.test.ts` use a `_streamText` stub that fakes multi-step behavior, so the test surface didn't catch the bug.

**Other R3 findings:**
- Middleware lacks `estimateTokens` fallback — safety-net silently no-ops on usage-less providers (Cloudflare Workers AI is a known one)
- Trace bridge is an in-memory array — production OTel/Langfuse observability is silently dropped
- Cap wrapper does not invoke `loadCapabilityOptional` (caller's job per README, but undocumented in adapter API)
- `runner.test.ts:128-201` detector test admits in plain English it tests the negative shape — title says "tests detector firing"; body and assertions don't

**R3 verdict:** "any framework runs in a pod" claim is now TRUE-WITH-FIVE-CAVEATS, **worse than rev2**. The reference adapter that was supposed to prove the claim survives v0.1.9 actually demonstrates it does NOT — the runner can't run a multi-step loop without a fix.

**R3 fix path:**
1. Pass `stopWhen: stepCountIs(opts.maxSteps ?? 16)` to `streamText`. ~3 LOC.
2. Add a regression test that runs the adapter against a fake provider returning multiple tool-call/result rounds, asserting the runner makes at least 2 LLM calls.
3. Wire `estimateTokens` fallback in the middleware for usage-less providers.
4. Replace the in-memory trace bridge with a TraceSink that delegates to the same OTel/Langfuse path `@kagent/agent-loop` uses.

**Fix:** `model-index.ts:147` — change `ep.spec.minSafe ?? 1` to `Math.max(1, ep.spec.minSafe ?? 1)`. Same fix at `admin-routes.ts:183`.

- NEW-H1 is MCALL (sibling pattern; required dep wired but a callsite is missing)

The paradigm doc captures these distinctions correctly via the WBD/MCALL/CSPREAD/DEADBRANCH taxonomy.
