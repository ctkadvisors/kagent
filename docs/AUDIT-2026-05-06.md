## Executive summary

This audit shoots holes in three claims:

1. **"Only OSS K3s-native agent framework"** — **FALSE as written.** Falsified by `kubernetes-sigs/agent-sandbox` (2,057★, May 2026) — the upstream Kubernetes SIG owns the per-agent isolation primitive — and by Solo.io kagent.dev (2,689★, CNCF Sandbox), which already integrates it. **Survives only as a narrowed claim around the capability sealed-JWT primitive, which nobody else ships in OSS as of 2026-05-06.**
2. **"Composable — any framework runs in a pod"** — **TRUE-WITH-CAVEATS.** Vercel AI SDK and Strands TS drop in cleanly with ~100 LOC of glue. Claude Agent SDK does NOT (Anthropic Commercial ToS + cannot route through LiteLLM). All third-party runtimes need per-framework cap-claim adapters because no framework other than the Claude Agent SDK has a `canUseTool` hook.
3. **"Falsifiable success — 5-topic researcher workload through both substrates for one week"** — **NOT MEASURED.** `docs/V0.1-COMPARISON.md` does not exist. `evidence/` contains `rc1/` and a `demo-success.png`. The project's own pre-committed success criterion has no data.

Code-side, the audit found **5 BLOCKERs** that should not ship:
- A verifier↔job-watch label collision (C1) that pollutes parent AgentTask conditions.
- A model-class doc/code mismatch (C1) — the docs claim hot-reload, the code requires an operator restart.
- A capability-bundle fail-OPEN by design (C2) — missing JWT mount silently disables every cap-gated guardrail.
- A JWKS-unreachable-at-boot path (C2) that CrashLoops without writing a `Failed` status.
- A workbench `minSafe=0` PATCH (C3) that lets any authenticated user permanently DoS a model endpoint, plus a header-spoof window (C3) that makes "authenticated" a movable trust boundary on default-flannel K3s clusters.

Net assessment: **the codebase is not a clown car.** The status-monotonicity tests, the AgentTask reconciler 409-handling, the modelClass resolver's loud failure mode, and the capability-narrowing wire-up are all carefully reasoned. But the substrate-as-shipped has authority and DoS gaps the docs do not warn about, AND the strategic framing has been overtaken by upstream events. Either narrow the claim, contribute upstream, or rename and reposition.

### R1.2 Verdict on the load-bearing claim

> "kagent is the only OSS K3s-native agent framework — composing per-agent-microVM substrate primitives that AWS AgentCore, Cloudflare Agents+Sandbox, and Anthropic Managed Agents all ship proprietarily."

**Status: FALSE as written. Has a defensible narrowing — but only one.**

Two converging facts falsify it:

1. **`kubernetes-sigs/agent-sandbox` (2,057★, May 2026)** ships `Sandbox`, `SandboxTemplate`, `SandboxClaim`, `SandboxWarmPool` CRDs with explicit Kata/gVisor RuntimeClass support. Apache-2.0. **The Kubernetes upstream owns the per-agent isolation primitive now.** Kubernetes blog formally blessed the pattern on 2026-03-20 (https://kubernetes.io/blog/2026/03/20/running-agents-on-kubernetes-with-agent-sandbox/).
2. **Solo.io kagent.dev (2,689★, CNCF Sandbox)** ships an `Agent` CRD plus a `SandboxAgent` CRD that delegates to agent-sandbox. They get isolation by composition. They have the kagent name in Google search results.

A reasonable engineer in May 2026 can build a 4-of-7 substrate today by composing: **(a)** kubernetes-sigs/agent-sandbox for primitives 1+4, **(b)** Solo.io kagent for primitive 1 declarative shape and (partial) 7, **(c)** Argo Workflows + Argo Events on NATS for primitives 2+3+7, **(d)** SPIFFE/SPIRE + Istio Ambient (Red Hat Kagenti's pattern) for Identity. They get Kata isolation, NATS event bus, SPIFFE identity, OTel observability. **They do NOT get capability sealed-JWT narrowing or content-addressed Artifact CRD or unified Audit-on-JetStream — that's the entire defensible delta as of 2026-05-06.**

### R1.3 Three possible re-positionings

**Option A (recommended): Narrow the claim.**
> "kagent is the first OSS K3s-native operator to compose per-agent isolation, durable orchestration, NATS JetStream A2A bus, and **sealed-JWT capability narrowing** into a single substrate. The capability primitive is unique; the others are integrations."

**Option B: Drop "only" entirely; compete on integration.**
> "kagent is an opinionated OSS bundle — what AgentCore/Anthropic/Cloudflare make you pay hosted, and what Solo.io kagent + agent-sandbox + Argo + SPIRE makes you assemble."

**Option C: Consolidate. Donate AgentTask / Workflow / Capability / Workspace / Artifact CRDs upstream** to kubernetes-sigs/agent-sandbox or Kagenti; reposition ctkadvisors/kagent as the homelab-flavored opinionated distro on top.

### R1.4 Top 5 surprises (R1)

1. **`SandboxAgent` exists in Solo.io kagent** (`go/api/v1alpha2/sandboxagent_types.go`) — i.e., the kagent.dev project has *already integrated kubernetes-sigs/agent-sandbox*. The composition kagent claims novelty for is shipped.
2. **Kubernetes blog (2026-03-20)** formally blessed the Sandbox CRD pattern. When the upstream blog calls something "the way to run agents," novelty for that primitive is gone.
3. **Red Hat Kagenti has the strongest identity story of anyone surveyed** — auto-injected SPIFFE SVIDs, x5c-signed AgentCards via SPIRE, Istio Ambient per-pod mTLS, Phoenix + OTel observability. kagent's planned SPIFFE primitive is not novel; consider just adopting Kagenti's pattern.
4. **Nobody other than kagent ships sealed-JWT capability tokens with caveat-narrowing on spawn.** Macaroons exist as a concept since 2014; Microsoft toolkit signs plugins with Ed25519; Kagenti uses Keycloak RFC-8693 token exchange. **None of these are JOSE ES256/RS256 capability bundles that narrow on spawn.** Defend this hill.
5. **Mastra has 23,618 stars and zero Kubernetes operator.** The OSS agent market doesn't actually demand a K8s substrate yet — it demands a TypeScript SDK. Marketing implication: target the platform-engineering-on-K8s crowd, not the agent-builder crowd.

### R2.2 Strongest counterclaim to kagent's pitch

**It is NOT AgentCore / Anthropic / Cloudflare** — none of those vendors ship on-prem. **It is the combination of (a) Cloudflare Project Think + Sandbox + Workflows v2 *as a shipped surface* and (b) LangGraph Platform / Temporal / Restate *as the self-host story*.**

- **AgentCore self-host:** NOT shipped. Only Identity-on-ECS allows on-prem agents to consume AgentCore Identity; control plane (Runtime, Memory, Gateway, Policy, Observability) is AWS-resident.
- **Anthropic Managed Agents on-prem:** NOT shipped. Cloud-only, Anthropic-data-path.
- **Cloudflare Agents on-prem:** NOT shipped. `workerd` runs locally but no production self-host story.
- **BUT Cloudflare has shipped *ten more substrate primitives in 2026* than kagent has even sketched** (Browser Run, Outbound Workers, Sandboxes GA, Workflows v2, AI Search, Voice Pipeline, Artifacts, Mesh, per-agent scheduling, OAuth-for-Access). The "we deliver in OSS what they sell hosted" framing is true for the 7 primitives kagent has *chosen*; it is **false for the surface area Cloudflare has actually shipped to compete with**.
- **LangGraph Platform self-host on K8s:** **SHIPPED on Enterprise plan, full Helm chart, no-data-leaves-VPC.** For the AgentWorkflow primitive specifically, LangChain has direct OSS-on-K8s competition with vastly more agentic mind-share.
- **Temporal Nexus + Cloud + self-host:** SHIPPED. $5B valuation, 380% YoY revenue. Temporal is what most enterprises will reach for first.

**Sharpest version:** kagent's defensible pitch is not "OSS substrate" — it is **"one orthogonal 7-primitive substrate vs. seven different point tools that don't compose."** Lead with composition.

### R2.3 Substrate-gap inventory (HIGH-leverage)

1. **A2A protocol conformance on the Event primitive.** Vertex / AgentCore / Foundry / 150+ orgs run A2A v1.2 in production. kagent's `docs/PRIOR-ART.md` and `SUBSTRATE-V1.md` do not mention A2A. **Highest-leverage gap.** Without it, kagent agents are an island.
2. **Per-AgentTask scheduled triggers (cron / delay / absolute)** — Cloudflare's `agent.schedule(when, methodName)`. kagent has webhook + scheduler entry-points (`v0.1.16`) as a *trigger system*, not a per-AgentTask self-scheduling primitive.
3. **Programmable egress proxy with credential injection (Outbound Workers analog).** kagent's `Capability.egress` claim → NetworkPolicy is allow-list domains. Cloudflare's pattern injects credentials into outbound calls inside the sandbox without exposing them to agent code — substantially stronger than env-var-passing.
4. **Sandbox snapshot + resume of FS+memory state.** Modal, OpenAI SandboxAgent, Cloudflare Sandboxes, AgentCore Code Interpreter all ship this.
5. **Cedar-style policy DSL alongside JWT capabilities.** AgentCore Policy lets ops teams author rules in a real language. kagent capability JWT claims are the right trust root; adding a Cedar-or-rego compiler that produces capability claims would give the same audit-able policy surface.

### R2.4 Top 5 surprises (R2)

1. **Cloudflare ships a primitive literally called "Artifacts"** (Git-compatible versioned storage) in 2026. The naming collision with kagent's `cas://sha256:` URI is total.
2. **Anthropic Managed Agents picked the exact triple "Agent / Environment / Session"** which maps 1:1 to kagent's Agent / Workspace / AgentTask. Independent convergence is *evidence in kagent's favor* that the 7-primitive shape is real — but Anthropic's docs are now the canonical shape kagent will be compared against.
3. **A2A protocol is at v1.2 with 150+ orgs in production, governed by the Linux Foundation.** kagent's substrate docs do not mention A2A at all. **Load-bearing omission.**
4. **AWS shipped Cedar policy enforcement at the Gateway tool-call layer.** kagent has neither a policy DSL nor an admission webhook for capability narrowing rules.
5. **Modal claims 100k concurrent sandboxes in production at one customer, target 1M.** kagent has zero published concurrency benchmarks. The "K3s-native per-agent microVM" pitch has zero benchmark backing — **the single most important credibility gap for the production-scale framing**.

### R3.2 Steel-man for "@kagent/agent-loop is a wasted abstraction"

If Vercel AI SDK can drop in with ~100 LOC of glue, AND it ships `AbortSignal.timeout()`, `stopWhen: stepCountIs(N)`, `experimental_telemetry` + OTel NodeSDK, `createGateway({baseURL})`, `tool({inputSchema, execute})`, with 24k stars / 4391 code snippets / weekly releases — then `@kagent/agent-loop`'s 3,000 lines of TS reproduces — almost line-for-line — primitives the AI SDK already gives you for free, with worse community mindshare and zero pluggable model providers.

**agent-loop earns its keep in exactly four narrow places**, all visible in the source tree:
1. **`detectors/` + `computeQualityFlags` / `detectRefusal`** — these aren't loop concerns; they're a kagent-substrate concern. Could live in `@kagent/run-detectors` consumed by ANY framework.
2. **HTTP-429 exponential-backoff retry policy (commit c74ab16)** — Vercel AI SDK doesn't have first-class 429 retry on the model layer. Earns its keep, but could be middleware not a whole executor.
3. **`registry.ts` (185 LOC) — AgentRegistry + skill-suitability scoring.** Routing concern, not agent loop concern. Doesn't belong in agent-loop at all.
4. **`TraceSink` interface** — could be a 30-LOC adapter on top of any framework's spans.

**Where it's pure overhead:** re-implementing tool-call dispatch, ChatMessage shapes, LLMClient adapters, OpenAI-compat protocol parsing. Vercel AI SDK does all of this with better OTel semconv coverage and a real provider ecosystem (Bedrock, Mistral, Cohere, Gemini, OpenRouter, Letta — kagent has zero of these today).

The "70% lifted from agent-runtime" framing reveals the trap: the lift was defensible at the prior project's scope. Inside kagent — which **already mandates LiteLLM** — agent-loop's core value prop (vendor-agnostic LLM routing) is double-billed.

### R3.3 Verdict on "any framework runs in a pod"

**TRUE-WITH-CAVEATS**, with three named friction points:

**Friction 1 — Substrate-supplied tools.** `spawn_child_task`, `wait_for_child_task`, `wait_for_children_all`, `ensure_agent_from_template`, the four blackboard tools, `publish_event`, `get_my_context` are wired via `InProcessToolProvider`. Each candidate framework has a different tool primitive. To make the "any framework" claim real, kagent needs **per-framework tool adapters** that re-emit the same nine substrate tools in each framework's vocabulary. ~200 LOC × N frameworks.

**Friction 2 — Capability JWT enforcement.** No third-party framework has a "consult an external claim object before letting the LLM call this tool" hook. Vercel AI SDK has no `canUseTool` callback. OpenAI Agents JS has handoffs but no per-call cap check. **Only Claude Agent SDK has `canUseTool: CanUseTool`** — and that's the framework whose license disqualifies it from being a default. So cap-narrowing is achievable per-framework only by wrapping each tool's `execute` with `if (!claim.allows(target)) return policyDeniedResult()`.

**Friction 3 — OTel `gen_ai.*` semconv consistency.** Each framework emits its own span shape; Langfuse traces will look different per framework unless kagent ships a normalizer.

**One-sentence verdict:** TRUE for any framework that exposes a `(prompt, tools[], abortSignal) → Promise<result>` boundary, which is most of them; FALSE in the "drop in cleanly" sense because (a) substrate tools require per-framework adapters, (b) capability enforcement requires per-framework tool-execute wrappers, (c) trace shape will diverge across frameworks, and (d) the headline 2026 candidate (Claude Agent SDK) is license-incompatible AND can't be routed through LiteLLM.

### R3.4 Top 5 surprises (R3)

1. **`@anthropic-ai/claude-agent-sdk` is not an in-process SDK** — it shells out to the `claude` Code CLI binary.
2. **Mastra's license is dual** — Apache 2.0 for most of the tree, but `packages/core/src/auth/ee/` is proprietary EE code. GitHub reports `NOASSERTION`.
3. **Strands TS only hit 1.0 in early 2026** and is still flagged "experimental, breaking changes expected." TS port is six months behind Python on features.
4. **OpenAI Agents JS has no native OTel `gen_ai.*` emission** — ships `OpenAITracingExporter` to OpenAI's traces dashboard.
5. **Agent Skills are not a framework** — they're `SKILL.md` directories loaded via filesystem progressive-disclosure inside Claude Code's VM.

## C1 — Operator correctness audit

> Net verdict: not a clown car — the dispatch path, status-monotonicity helpers (`patchStatusWithRetry` / `nextPhase` / `mergeCondition`), and 409-AlreadyExists handling are well thought out, and the test surface is real (~17K LOC test code vs ~24K LOC source). But there are several real correctness defects, one of which is a load-bearing label collision between verifier Jobs and the agent-Job watcher, plus the doc-claimed "kubectl-friendly hot-reload" of model classes that doesn't exist.

### C1.1 BLOCKER

- **[BLOCKER] [code-correctness]** Verifier Jobs share the same `managed-by=kagent-operator` + `kagent.knuteson.io/task=<parent>` labels as the dispatch Job, so verifier failures pollute the parent AgentTask.
  - `packages/operator/src/verifier.ts:721-725` (Job labels), `:744-746` (Pod template), vs `packages/operator/src/job-watch.ts:34-35` (`MANAGED_BY_LABEL` + `TASK_LABEL` selector).
  - The verifier defines a discriminator label `VERIFIER_JOB_LABEL = 'kagent.knuteson.io/verifier'` (`verifier.ts:124`) "so the operator (and future GC sweeps) can filter them out of the regular agent-Job stream" — but `job-watch.ts` and `failure-detector.ts` **never consult it**. When a verifier Job exits non-zero (i.e. on `verdict:fail`, the entire point of script verifiers), the parent AgentTask is in `Completed`, so phase isn't clobbered — but a `JobFailedAfterComplete` condition with `reason=JobFailed`/`exit code 1` is appended to `status.conditions[]` on every relist forever.
  - Smallest fix: in `job-watch.ts`, drop events whose `metadata.labels[VERIFIER_JOB_LABEL] === 'true'` before calling `parentTaskRef`. One-line guard, zero RBAC change.

- **[BLOCKER] [doc-mismatch + code-correctness]** Model-class map is a one-shot env-JSON parse at boot, not a hot-reloadable ConfigMap. Cluster-side updates require restarting the operator.
  - `packages/operator/src/main.ts:194-223` (`parseModelClassesEnv`), `:507` (called once in `buildJobSpecOptionsFromEnv`), chart at `packages/operator/charts/kagent-operator/templates/deployment.yaml:268-269` (`KAGENT_AGENT_MODEL_CLASSES_JSON: {{ .Values.agent.modelClasses | toJson | quote }}`).

  - In-flight pods continue with the physical model id baked into their `agent.spec.json` ConfigMap at dispatch-time, so they don't migrate mid-flight (correct), but new tasks for a removed class fail with `unresolvable_model_class:`.
  - Smallest fix: ship a chart ConfigMap + an operator-side ConfigMap watcher that swaps the map reference in place, OR update `MODEL-ROUTING.md` to delete the hot-reload claim. Pick one.

## C2 — Agent-pod boot + spec injection audit

### C2.1 BLOCKER

- **[BLOCKER] [cap-consumer]** Capability bundle is fail-OPEN by design — JWT mount absence silently disables every cap-gated guardrail in the substrate.
  - `packages/agent-pod/src/cap-consumer.ts:148-167` (`loadCapabilityOptional`): on ENOENT returns `undefined` and the pod logs "capability bundle absent; running legacy claim path."
  - `main.ts:117-125` calls `loadCapabilityOptional` (not `loadCapabilityFromEnv`). Once `capabilityBundle === undefined`: `defineSpawnChildTask` falls back to `allowedChildAgents` (which most operator-facing Agent CRs don't set anymore — see C2.3 spawn finding), `defineGetMyContext` strips its `capability` field, `definePublishEvent` refuses ALL emissions, and the `claims.tenant` header is omitted from every gateway call so per-tenant routing/quota silently regresses to "no tenant."
  - **Threat model:** an attacker (or simply an upgrade error) who suppresses, deletes, or forgets to mount the Secret gets a pod with zero in-pod authority enforcement. The `[kagent-agent-pod] capability bundle absent` log line is the only signal.
  - WAVES.md §4.1 calls the JWT mount "load-bearing"; the code makes it advisory.
  - Smallest fix: gate fail-open behind `agentPod.capability.allowMissing: false` (default), and refuse to boot if `capabilityBundle === undefined` AND `KAGENT_CAP_JWT_FILE` was explicitly stamped on env.

- **[BLOCKER] [cap-consumer]** JWKS fetch failure at boot is treated as terminal — but the fetch happens on EVERY pod start with no cache, so a flaky operator-side template-server kills every spawned pod.
  - `cap-consumer.ts:121-134`: no retry, no in-process cache, no fallback issuer config. `defaultFetchJwks` (`:196-210`) does single `fetch(url)` with no timeout, no retry, no exponential backoff.
  - If `kagent-template.kagent-system.svc:8081/.well-known/jwks.json` is briefly unreachable (operator pod restarting, NetworkPolicy hiccup), the agent pod throws inside `loadCapabilityFromEnv` — bubbles out of `loadCapabilityOptional`'s body, the pod crashes, the Job goes CrashLoopBackoff, and the AgentTask sits in `Dispatched` forever because `main.ts` never gets to write a status patch.
  - Smallest fix: add 10s timeout + 3-attempt exponential retry around `defaultFetchJwks`; on terminal JWKS failure write a `Failed` status patch with `error: jwks_unreachable: <detail>` BEFORE exiting.

- **[BLOCKER] [status]** `parentDistillation` env-JSON path has no size validation — single oversized field can blow past ARG_MAX before the Job starts.
  - `env.ts:324-332` accepts the deprecated field with only a warn. Combined with env-JSON fallback path still being live (`:466-491`), an oversized `parentDistillation` shipped via env can exceed Linux ARG_MAX. Pod fails to *start* with `argument list too long`; operator job-watch sees `Failed` Pod with opaque kubelet error; AgentTask gets generic `Failed` with no useful `error:` reason because `main.ts` never ran.
  - Smallest fix: in `env.ts:324-332`, refuse the parse with explicit size cap; gate env-JSON path behind hard 256 KiB cap on combined `KAGENT_AGENT_SPEC + KAGENT_TASK_SPEC` payload.

### C2.2 HIGH

- **[HIGH] [tool-resolution]** `Agent.spec.tools` allowlist is enforced at boot but executor's `providerFor` lookup is GLOBAL — providers can claim tools the spec didn't list and the LLM call succeeds.
  - `runner.ts:531-565` (`resolveToolProviders`) builds the `builtin` provider from `config.agentSpec.tools`, then UNCONDITIONALLY appends `spawnTools` / `blackboardTools` / `eventsTools` whenever env-flags fire. `executor.ts:708`: `const provider = this.toolProviders.providerFor(toolCall.name);` looks across the FULL federation registry. **No cross-check that the resolved provider's tool name was in `Agent.spec.tools`.**
  - An Agent CR that does NOT list `spawn_child_task` in `spec.tools` but is deployed in a cluster where `KAGENT_SPAWN_CHILD_ENABLED=true` ends up with the spawn tool exposed to the LLM.
  - Smallest fix: in `runner.ts:resolveToolProviders`, after appending substrate providers, assert every tool name they expose is also in `config.agentSpec.tools`.

- **[HIGH] [cap-consumer/publish]** `definePublishEvent` falls back to a SELF-MINTED capability bundle when no JWT is mounted — bypassing JWKS verification entirely for the publish path.

- **[HIGH] [status]** Status writer uses `merge-patch+json` but never sets `resourceVersion` — last-writer-wins races between `main.ts` and operator job-watch can clobber `Completed` with `Failed`.
  - `status.ts:30,131-141`. JSON merge-patch on the status subresource has no optimistic concurrency.
  - Smallest fix: build the patch as JSON Patch with a `test` op asserting `status.phase` is `Pending` or `Dispatched` (refusing to overwrite a terminal phase).

- **[HIGH] [executor/retry-storm]** `wait_for_children_all` polling loop AND the 429-retry policy are independent — a 429 storm during the parent's wait can blow past the parent's wall-clock budget.
  - `executor.ts:296-302` (`pickBackoffMs`): `Retry-After` is honored verbatim with no clamp against the run's remaining wall-clock budget. ### C2.5 The env-JSON cap audit

**Cap math.** Linux kernel default `ARG_MAX` is 2 MiB shared by env block + argv. K8s adds its own ceilings: per-env-var value bounded by etcd's per-object 1 MiB practical write limit, kubelet writes env block into container process memory at exec — observable via `/proc/<pid>/environ`.

**Combined caps for env-JSON path (when `useConfigMap=false`):**
- `KAGENT_AGENT_SPEC` = `JSON.stringify(resolvedAgentSpec)`.
- `KAGENT_TASK_SPEC` = `JSON.stringify(task.spec)`.
- ~30 small operational env vars (~4 KiB total).
- Init container env, K8s-injected (~1 KiB).

**Practical hard cap before kubelet refuses to spawn:** ~1 MiB on the etcd side; ~500 KiB on the kernel side.

**Which `AgentSpecEnv` fields plausibly push past 256 KiB:**
1. **`agentSpec.systemPrompt` literal.** No length cap anywhere. A long structured-output system prompt with examples (50-100 KiB) is realistic.
2. **`agentSpec.publishes[].schema` and `.subscribes[].schema`.** JSON Schema documents at 5-50 KiB each. 10 publish topics × 10 KiB = 100 KiB.
3. **`taskSpec.payload`.** Typed `unknown`; validated nowhere on the agent-pod side.
4. **`taskSpec.parentDistillation`** — deprecated but accepted, no size validation in `env.ts:324-332`.
5. **`agentSpec.capabilityClaims`.** A deeply nested or array-heavy `claims.spawn: [<thousands of entries>]` ships verbatim.

**Concrete failure mode:** Job goes `BackoffLimitExceeded`; Pod creation fails with `argument list too long` in kubelet event log; operator job-watch surfaces `Failed` with no agent-pod `error:` reason.

**Recommendation pre-cutover:** Stamp a hard 192 KiB combined cap at `job-spec.ts:666-671` BEFORE serializing into env. Then the deprecation warning in `env.ts:328` becomes a hard error rather than silent corruption.

### C2.6 Top 3 risks for an enterprise consumer dropping non-`@kagent/agent-loop` runtime into a pod

1. **Status writeback contract is implicit and undocumented at the pod boundary.** A drop-in runtime that writes its own status patch will land an AgentTask the operator can never reconcile cleanly (must use merge-patch, must nest under `result.content`, must include `structuralVerdict.suspicious` even when empty).
2. **Capability-bundle threading is invisible to non-`@kagent/agent-pod` runtimes.** A drop-in runtime gets `KAGENT_CAP_JWT_FILE` and `KAGENT_CAP_JWKS_URL` env vars but nothing else. Must read JWT from file (NOT env), fetch JWKS, verify against ES256/RS256 only, thread `claims.tenant` into every LLM call as `X-Kagent-Tenant`, thread `claims.publish` into every event emission, thread `claims.spawn` into every K8s create call. Skipping any breaks the multi-tenant story silently.
3. **The "tool resolution = `Agent.spec.tools` allowlist" contract is enforced ONLY by the agent-pod's `resolveBuiltinTools` — the executor itself dispatches whatever the LLM names against whatever providers are registered.** The substrate ships *tool names* but not *tool admission policy*. There is no central admission point.

The substrate needs an **agent-pod-runtime conformance test suite** (cap mount → claim threading → status patch shape → tool-name vs spec.tools cross-check) before the "any in-pod runtime" pitch is honest.

## C3 — Workbench, gateway, security boundary audit

### C3.1 BLOCKER

- **[BLOCKER] [auth/dos]** PATCH `/api/modelendpoints` permits `minSafe: 0`, allowing a single error to drive AIMD `cap` to 0 → permanent DoS for that model.
  - `packages/workbench-api/src/routes/gateway.ts:90` declares `MIN_SAFE_MIN = 0`; validation at L196-207 admits `minSafe: 0`.
  - `packages/llm-gateway/src/aimd.ts:139,157` halves `cap` on latency-spike or `onError` clamped at `state.bounds.minSafe`. With `minSafe = 0` and `cap = 1`, a single 5xx → `Math.max(0, Math.floor(1/2)) = 0`. Subsequent `route()` hits `inFlight (0) >= cap (0)` at `router.ts:129` and returns 429 forever; `aimd.onSuccess` is never reached because no request ever clears the cap-check, so additive-increase at `aimd.ts:144` never fires. Persists until operator restart or hand-edit.
  - **Attacker scenario:** any authenticated workbench user PATCHes `{minSafe: 0}` on the cluster's primary model endpoint, induces (or waits for) one transient 502, and the model is permanently 429'd.
  - Smallest fix: raise `MIN_SAFE_MIN` to 1.

- **[BLOCKER] [auth]** Direct in-cluster requests to `workbench-api` Service can spoof `X-Forwarded-User` because no NetworkPolicy or chart template restricts ingress to the Traefik forward-auth path.
  - `auth.ts:75-106` trusts `X-Forwarded-User` verbatim if non-empty.
  - `kagent-workbench/templates/networkpolicy.yaml:48-60` declares `ingressFrom` defaults to a `namespaceSelector` for `ingress-controller` namespace ONLY when not overridden; chart's `values.yaml:265,267 ingressFrom: []` ships empty. **Chart README acknowledges in the same file (L18) that "K3s with the default flannel CNI does NOT enforce NetworkPolicies."** Worse, `templates/ingress.yaml:35` only chains an auth Middleware when `ingress.authMiddleware` is non-empty (default: `''`); the plain-Ingress render path (L66-98) emits NO auth at all.
  - **Attacker scenario:** any pod in any namespace on a default flannel K3s install issues `curl -H 'X-Forwarded-User: anyone' -X POST .../api/tasks` and creates AgentTasks (`actions.create=true` is default per `values.yaml:181`). Combined with cluster-scoped actions ClusterRole, this is privilege escalation from any tenant pod to "create AgentTasks anywhere."
  - Smallest fix: make chart `fail` Helm install if `ingress.authMiddleware` is empty AND `WORKBENCH_AUTH_REQUIRED != "false"`.

- **[BLOCKER] [secret-hygiene]** Bundled-Postgres Secret writes the DSN with the password in cleartext into a non-Sealed Helm-managed Secret keyed under `dsn:`, and Helm's `lookup` round-trip pattern leaks the password via `helm template` / `helm get manifest`.
  - `packages/llm-gateway/charts/llm-gateway/templates/secret-bundled.yaml:33-43`: `$password` read via `lookup` (or generated fresh on first install) concatenated into a postgres URI under `stringData.dsn`. Anyone with `get secret` RBAC in the release namespace can read cleartext password. `sslmode=disable` (L43) compounds.
  - Smallest fix: move bundled-Postgres to `sslmode=verify-full`; document Sealed-Secrets / external-secrets as production path; mount user/password separately via `secretKeyRef`.

### C3.2 HIGH

- **[HIGH] [dos]** Gateway never propagates upstream backend `Retry-After` to the agent-pod — only its OWN AIMD-generated 429 carries `Retry-After: 5`. A real 429 from OpenAI/Anthropic/Cloudflare lands as `kind: 'dispatch_error'` (HTTP 502 with no Retry-After), so the agent-pod retries IMMEDIATELY and stampedes the upstream.
- **[HIGH] [auth/crypto]** Admin-token comparison length-leaks via the `length !== expectedToken.length` early return BEFORE `timingSafeEqual` (`admin-routes.ts:38-40`). The whole point of `timingSafeEqual` is defeated. Use HMAC-SHA256 digests of both sides for constant-length comparison.
- **[HIGH] [secret-hygiene]** Provider error responses propagate upstream response bodies VERBATIM through to `usage_records.error_message`, surfacing in `/admin/usage`. A misbehaving backend echoing the prompt → trace exposes PII to anyone with admin token.
- **[HIGH] [input-validation]** Workbench `POST /api/tasks` `payload` field is wholly opaque `unknown` with no size cap or shape check. 100 AgentTasks × 1MB `payload` = 100MB written to etcd. Add `MAX_PAYLOAD_BYTES = 65_536` check.
- **[HIGH] [rbac]** Workbench actions ClusterRole grants cluster-scoped `agenttasks: [create]` (`clusterrole-actions.yaml:33`). Combined with spoofed `X-Forwarded-User`, any caller can create AgentTasks in ANY namespace by setting `req.namespace`. Scope to a Role in the operator's release namespace.
- **[HIGH] [auth]** API key authentication TOUCH path (`api_keys.last_used_at`) is documented but never invoked — every request bypasses last_used_at tracking. Rotation policy can't safely revoke "stale, unused for 30d" keys.
- **[HIGH] [auth]** llm-gateway conformance harness for SVID/mTLS fallback is mocked-only — `evaluateMtlsSvidFallback` reads result from an INPUT struct, not a live probe. The GATEWAY-CONTRACT.md §14 claim that "the conformance harness verifies… mTLS fallback" is currently false — it verifies the CLI invoker's claim about mTLS fallback.
- **[HIGH] [crypto]** Cap-CA `detectAlgFromPem` distinguishes ES256 vs RS256 via a length heuristic on the base64 body — not via ASN.1 parse. Risk surface small (Secret is chart-owned), but the heuristic is fragile.

### C3.3 MEDIUM

- `POST /api/tasks` accepts caller-supplied `name` without prefixing — caller can name their task `argocd-app-of-apps`. UI-listing impersonation, audit-grep collision.
- `/api/cluster/snapshot` issues uncached cluster-wide `listNode()` on every call; no rate-limiting on any workbench-api endpoint.
- `parsePatchInFlightBody` defensible but brittle (`Number.isInteger` ordering).
- `/api/gateway/usage` proxies gateway `error_message` rows verbatim — combined with HIGH #3 above, leaked upstream content surfaces to authenticated workbench users without scrubbing.
- Admin token held in plaintext in workbench-api process memory for the process's lifetime.
- `/api/stream` (SSE) has no per-client connection cap or per-user rate limit.
- `WORKBENCH_AUTH_REQUIRED` is configurable via Helm and a YAML truthy/string mismatch could disable auth (`api.authRequired: false` as YAML boolean → `value: "false"`).
- JTI uses 4 bytes (~32 bits) of entropy — insufficient if downstream consumers treat it as cryptographically opaque.
- `parseRevokeIdFromUrl` does not validate `id` is numeric — surfaces Postgres error text.

### C3.4 LOW / NIT

- `validators.ts` accepts single-char target agent names.
- `KEY_PREFIX_LEN = 8` includes the literal `sk-` prefix, leaving 5 chars of entropy.
- `bedrock-provider.ts` is a stub that THROWS on call — should be admission-rejected at operator's `ModelEndpoint` admission controller.
- `headers.ts:34-55 parseKagentHeaders` reads `X-Kagent-Task-UID` / `X-Kagent-Agent` from EVERY caller and writes to `usage_records` — attribution is lieable.
- `pgPool` default `max: 10` plus single-replica gateway means 10 concurrent admin queries can starve the chat path.
- `tasks.ts:262` returns `K8s API call failed: ${err.message}` directly to caller — minor info disclosure.
- `cluster.ts:303 lastResultPreview` returns 200 chars of arbitrary task output to any authenticated viewer.

### C3.5 The "duplicate @cf/meta row" root cause

After auditing `model-index.ts`, `db/usage.ts`, `migrations/*.sql`, and `aimd.ts`, **the duplicate row is NOT a code bug at the model-index or capacity-row level** — both `ModelIndex` (`Map<spec.model, ModelEndpoint>`) and `AimdController` (`Map<\`${model}|${endpoint}\`, ...>`) key uniquely.

**The actual mechanism is a CR-level / informer-level duplication:**

1. `model-index.ts:36-38 upsert(endpoint)` keys by `endpoint.spec.model`. If TWO ModelEndpoint CRs declare the same `spec.model` (one in `kagent-system/cf-llama-scout`, one in `seekarc/cf-llama-scout`), the watch loop's last-write-wins `upsert` keeps ONE row in the gateway's index — but `model-watch.ts:80-88 apply(ep)` ALSO calls `aimd.updateBounds(model, backendUrl, ...)` for BOTH events. Same model, DIFFERENT `backendUrl`, AIMD's key is `${model}|${endpoint}`. Result: AIMD has 2 entries; ModelIndex has 1.
2. `buildCapacityResponse` (`admin-routes.ts:78-95`) iterates `modelIndex.list()` (1 row). No doubling.
3. **HOWEVER**, the workbench-api join at `gateway.ts:108-142 buildModelEndpointIndex` lists ALL `ModelEndpoint` CRs cluster-wide. Two CRs with the same `spec.model + spec.backendUrl` produce two index entries; `if (!index.has(key)) index.set(key, ...);` keeps the FIRST CR by iteration order (server-side returns no stable order). The second is silently dropped.

**The visible "duplicate row"** is most likely two `ModelEndpoint` CRs with `spec.model = '@cf/meta/llama-3.1-8b-instruct'`, DIFFERENT `spec.backendUrl` (one for AI Gateway, one direct), so AIMD has 2 entries → `aimd.snapshot()` returns 2 rows → but `buildCapacityResponse` doesn't iterate AIMD, it iterates ModelIndex which only has 1.

**The fix is admission-side**: the `ModelEndpoint` CRD validating webhook (`packages/operator/src/admission.ts`) should reject a CR whose `(spec.model, spec.backendUrl)` collides with an existing CR. There is no evidence in the code I read that this admission rule exists.

**Three concrete mitigations (priority order):**

1. **Make the chart fail Helm install when the auth posture is unsafe.** Add a `validateValues` template that requires `(ingress.authMiddleware != "" || ingress.enabled == false)` AND `api.authRequired == "true"` AND `networkPolicy.enabled == true`.
2. **Don't trust the header from anywhere except localhost / a verified upstream.** Switch from "header-trust" to verifying a cookie/JWT issued by oauth2-proxy.
3. **Tighten the actions ClusterRole to a Role** in the operator's namespace.

### C3.7 Top 3 risks for an enterprise pilot deploy

1. **The default chart install is unsafely open by default.** With `actions.create=true` (default), `WORKBENCH_AUTH_REQUIRED=true` (correct fail-closed), but `ingress.authMiddleware=''` AND `networkPolicy.ingressFrom=[]` AND a flannel CNI, an operator who runs `helm install` without overrides ships a workbench whose `X-Forwarded-User`-trusting endpoints are reachable by every pod in the cluster.
2. **Gateway capacity can be DoS'd via a single PATCH.** Until `MIN_SAFE_MIN` is raised from 0 to 1, any workbench user with PATCH access can lock the cluster out of a model until the operator manually restarts the gateway pod.
3. **Secrets-hygiene posture for the bundled-Postgres path is incomplete.** Cleartext password in plain Helm-managed Secret with `sslmode=disable`; combined with HIGH #3 (upstream-echoed content in `error_message`), bundled-Postgres becomes a high-value target.

## Appendix A — Audit answers to the 5 questions

### 1. Is kagent still the only OSS K3s-native agent framework as of 2026-05-06?

**No — qualified.** Two converging facts falsify it:
- `kubernetes-sigs/agent-sandbox` (2,057★, May 2026, Apache-2.0) ships per-agent isolation as a SIG project with explicit Kata/RuntimeClass support.
- Solo.io kagent.dev (2,689★, CNCF Sandbox) ships `Agent` + `SandboxAgent` CRDs that delegate to agent-sandbox.

A reasonable engineer in May 2026 can compose 4-of-7 substrate primitives today by pairing agent-sandbox + Solo.io kagent + Argo Events on NATS + Red Hat Kagenti's SPIFFE pattern. The defensible delta is **capability sealed-JWT narrowing** — nobody else ships it in OSS — and the unified CAS Artifact + Audit-on-JetStream composition.

### 2. Is it actually flexible? (Specifically: can you change runtime behavior without a git commit?)

**Partially.** The substrate has earned tier:
- **Models:** `Agent.spec.modelClass` (logical tier) decouples Agent CRs from physical models — change one chart-value to swap a tier. **But the docs claim hot-reload of the class map; the code requires an operator pod restart** (BLOCKER C1 #2).
- **Tools:** `Agent.spec.tools` is a hard allowlist enforced at agent-pod boot (unknown names fail FAST). **But the runner appends substrate tools (`spawn_child_task`, etc.) without cross-checking against `spec.tools`** (HIGH C2.2 finding) — tools the LLM can call exceed what the spec declared.
- **Children:** `allowedChildAgents` + `allowedChildTemplates` are GitOps-controlled. Capability claim `spawn: ['*']` BYPASSES these (MEDIUM C2.3 finding).
- **Workflows:** AgentWorkflow CRD shipped (`v0.3.2-workflows`); workflow handler is in a Deployment image — runtime change requires Deployment rollout, no live editing.
- **Schedules / Webhooks:** `KagentSchedule` CRD + HMAC webhook receiver are CRD-driven; new triggers without code change.

Net: **most legitimate "agent does new work" flows can happen without a git commit**, but several substrate-level things (model-class map, tool list per Agent) require chart redeploy or operator pod restart. The flexibility surface is real but uneven.

### 3. Is it actually composable? (Can you swap @kagent/agent-loop for Strands or Mastra without forking the agent-pod image?)

**No today; partially achievable with focused work.** Per R3 + C2.6:
- The substrate-supplied tools (9 of them) are wired via `InProcessToolProvider`. Each candidate framework has a different tool primitive — adapters per framework are mandatory, ~200 LOC each.
- Capability JWT enforcement requires per-framework wrappers around tool `execute` — only the Claude Agent SDK has a `canUseTool` hook, and the Claude Agent SDK is license-incompatible (Anthropic Commercial ToS) AND can't route through LiteLLM.
- Status writeback contract is implicit and undocumented at the pod boundary (must use merge-patch, must nest under `result.content`, must include `structuralVerdict.suspicious`).
- OTel trace span shape diverges per framework.

A Vercel AI SDK or Strands TS adapter is **achievable in ~100-200 LOC** but does not exist in the repo. The "any framework runs in a pod" claim should be qualified to **"any framework with `(prompt, tools[], abortSignal) → Promise<result>` shape, given a kagent-supplied per-framework adapter."**

### 4. Is it "k3s deploy on config change-less"?

**Mixed:**

| Configuration | Hot-reload | Pod restart | Chart redeploy |
|---|---|---|---|
| `agent.modelClasses` (chart values) | ❌ (despite doc claim) | ❌ | ✅ (operator pod) |
| Agent CR `spec.systemPrompt` | ❌ | ✅ (next AgentTask spawns fresh pod) | — |
| Agent CR `spec.tools` | ❌ | ✅ (per-pod boot) | — |
| Agent CR `spec.modelClass` | ❌ | ✅ (per-pod boot) | — |
| `ModelEndpoint.spec.inFlight` | ✅ (gateway watches) | — | — |
| Helm values for operator | ❌ | ❌ | ✅ |
| New Agent CR | ✅ (admission picks up) | — | — |
| New AgentTask CR | ✅ | — | — |
| KagentSchedule CR | ✅ | — | — |
| `agentPod.runtimeClasses.{default,strict}` (chart) | ❌ | ❌ | ✅ |

Verdict: **CR-level hot-reload is real**, **chart-values "hot-reload" is mostly fictional** — `modelClasses` is the headline doc-mismatch. The ROADMAP §4.x bullet about ConfigMap-mounted `modelClasses` was designed but not built.

### 5. The single highest-leverage fix that would most narrow the gap between pitch and current state

**Fix the capability authority story end-to-end.** Three changes that compose:

1. **Remove the synthetic-bundle fallback in `agent-pod/src/main.ts:434-451`** (C2 BLOCKER #2 underbelly). Today, `Agent.spec.capabilityClaims.publish` is silently trusted as if it were operator-signed. Delete the fallback; refuse `publish_event` with `policy_denied:no_capability` when no JWT is mounted.
2. **Make capability mount required-by-default in the chart** (C2 BLOCKER #1). Gate fail-open behind `agentPod.capability.allowMissing: false` (default); refuse to boot when JWT path was stamped but file missing.
3. **Cross-check tool registration against `Agent.spec.tools` at runner-boot** (C2 HIGH #1). Substrate tools should not appear in the LLM's tool list unless the Agent declared the corresponding intent (e.g., `allowedChildAgents.length > 0` ⇒ `spawn_child_task` admitted; `capabilityClaims.publish` non-empty ⇒ `publish_event` admitted).

These three changes together turn the kagent capability primitive from a "nominally shipped, bypassable in three ways" feature into a real authority root. **That is the one substrate primitive R1 found nobody else has shipped in OSS.** Without these fixes, the differentiating claim is empty marketing. With them, kagent has one defensible thing the upstream Kubernetes SIG, Solo.io kagent, and every framework SDK don't.

## Appendix B — Most surprising findings (5 bullets)

1. **kubernetes-sigs/agent-sandbox is real, in upstream Kubernetes, with an official blog post (2026-03-20).** kagent's "first / only" framing was overtaken by the SIG. Solo.io kagent.dev already integrates it. The naming-collision conversation is now an absorption-risk conversation.
2. **The capability bundle is fail-OPEN by design** (C2 BLOCKER #1) AND **the publish-event tool falls back to a self-minted unverified bundle** (C2 HIGH #2). The substrate's whole authority story rests on JWT signature verification — and there are two distinct paths around it that both look like "defense in depth" in the source.
3. **The operator's coverage gate is literally `0`** (`packages/operator/vitest.config.ts:16-21`) despite README/CLAUDE.md claiming ≥85%. The actual coverage is high in practice (17K LOC tests vs 24K LOC source) — but a future regression lands silently. Combined with zero direct tests for `watch.ts` / `job-watch.ts`, the highest-leverage uncovered code is precisely where the verifier-label-collision BLOCKER lurked.
4. **The Anthropic Managed Agents primitive triple is "Agent / Environment / Session"** — independent convergence on the same substrate decomposition kagent chose. That's *evidence in kagent's favor* the 7-primitive shape is real, but it also means Anthropic's docs are now the canonical shape kagent will be compared against by enterprise buyers. "Environment" is arguably cleaner than "Workspace."
5. **A2A protocol v1.2 is in production at 150+ orgs, governed by the Linux Foundation** — and kagent's `docs/PRIOR-ART.md` and `SUBSTRATE-V1.md` do not mention A2A at all. kagent's Event primitive could *be* A2A on the wire and would otherwise be incompatible with everyone else's agents. The single highest-leverage strategic gap.
