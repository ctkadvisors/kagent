## 2. The 5-topic researcher workload

The `homelab-orchestrator` repo today has **3 active topics** in [`topics.yaml`](../../homelab-orchestrator/topics.yaml) plus the commented-out `paperclip-mode` demo topic. The comparison rig needs five so the per-topic mean is not dominated by a single tail and so the workload exercises both lightweight and heavy paths.

The recommended 5-topic set (user can substitute any topic that both substrates can ingest):

| # | slug | shape | est tokens/run | rationale |
|---|---|---|---|---|
| 1 | `cloudflare-workers-ai` | RSS-feed scrape + summarize | ~6-8k | existing topic; lightweight, well-defined sources |
| 2 | `mcp-spec-activity` | atom + commit feeds | ~6-10k | existing topic; tests `rss_fetch` + `extract_text` happy path |
| 3 | `anthropic-platform-activity` | mixed atom + HTML page | ~8-12k | existing topic; HTML page tests `http_get` + `extract_text` |
| 4 | `k8s-operator-design-activity` | new topic; survey of operator-pattern OSS commits | ~10-15k | tests larger context windows; relevant to substrate domain |
| 5 | `paperclip-mode` (re-enabled) | unbounded sources | 30-100k | tests context-pressure handling explicitly; deliberately stresses `KAGENT_CONTEXT_SAFETY_THRESHOLD` |

Topic 4 is new and exists to give the rig a topic both substrates can run that doesn't already exist as a homelab-orchestrator-tuned workload. Sources for topic 4 are operator-domain RSS / atom feeds (one of:
`https://kubernetes.io/feed.xml`, `https://github.com/kubernetes-sigs/agent-sandbox/commits/main.atom`,
`https://github.com/kagenti/kagenti/commits/main.atom`) — kept narrow on purpose so the comparison is between substrates, not between source-discovery quality.

Topic 5 (`paperclip-mode`) is the only one specifically chosen for stress: per [`HARNESS-LESSONS.md`](./HARNESS-LESSONS.md) it is the M2 enforcement regression target. On kagent it tests:

- the `context_pressure_ignored` detector (commit `fc32b13`)
- the substrate context-window safety-net at 95% (commit `73f67f4`)
- the `get_my_context` tool path (commit `fb549c0`)

On `homelab-orchestrator` it tests nothing equivalent; its purpose there is to expose the failure mode in baseline.

**Substitution policy.** The user may swap any topic. The constraint is: each topic must be representable as a researcher-style "fetch sources → extract text → synthesize digest" workload because both substrates support that shape and nothing else portably. Topics that need external API auth, Bedrock-only tools, or human approval gates are out.

## 3. The two substrates

### 3.1 kagent v0.1.x (current HEAD)

- **Shape:** `AgentTask` CR per topic per day → operator builds Job (or uses agent-sandbox path once that lands) → agent-pod runs `@kagent/agent-loop` against the configured LLM gateway → status patched with `result.content` + `structuralVerdict.suspicious[]` + trace link.
- **Inputs:** topic submitted via Workbench "New Task" (WS-J `v0.0.7-write-surface` shipped) or `kubectl apply` of a generated AgentTask manifest.
- **Tools:** `http_get`, `rss_fetch`, `extract_text`, `write_artifact`, `get_my_context` (built-ins, all shipped per Phase 5 / Phase 5.x).
- **Trace sink:** Langfuse (deep-link via `traceLink()` in `@kagent/dto`); JSONL fallback when Langfuse is offline.
- **Cost source:** LiteLLM Proxy if deployed (defer `v0.2`); otherwise the gateway's per-request `usage` block summed by `OtelTraceSink` into Langfuse `cost_details`.
- **Audit stream:** `v0.1.15-audit-stream` — CloudEvents on NATS JetStream, one event per dispatch / per child / per supervision action / per verifier verdict (47 event types per ROADMAP Wave 0).

### 3.2 homelab-orchestrator baseline

- **Shape:** Kubernetes CronJob fires at 06:00 → single `Node`/`tsx` process reads [`topics.yaml`](../../homelab-orchestrator/topics.yaml) → for each topic, builds an `AgentExecutor` (the pre-lift version of what is now `@kagent/agent-loop`) → writes `/digests/${slug}-${YYYYMMDD}.md` to a PVC + `/runs/${slug}-${YYYYMMDD}.jsonl` (full trace) to a sibling PVC.
- **Inputs:** topics are statically declared in `topics.yaml`; CronJob has no per-run knob.
- **Tools:** the same `agent-runtime` tool providers (HTTP fetch, RSS fetch, extract-text) but in-process, no built-in framework.
- **Trace sink:** JSONL on PVC; viewer at `orchestrator.knuteson.io` (now retiring per `WHY.md §5`).
- **Cost source:** computed in-process by `agent-runtime`'s `RunBudget` accumulator; surfaced in the JSONL trace's per-iteration cost field. No Langfuse, no LiteLLM.
- **Failure handling:** none beyond the CronJob's `backoffLimit`. F1/F2/F3 detectors run at run-end and stamp the JSONL but do not stop the run.

### 3.3 What stays the same

To make the comparison fair, both runs must use:
- The **same model and same gateway URL.** Recommended: Cloudflare AI Gateway → `workers-ai/@cf/meta/llama-4-scout-17b-16e-instruct` (or the Llama-3.3-70B routing if Scout's tool-call shape regresses, per `HARNESS-LESSONS.md`).
- The **same temperature / topP / maxTokens.** `Agent.spec.llmParams` (`v0.1.4-llm-params` shipped) → set the same values on the homelab-orchestrator `runConfig`.
- The **same five topic strings** (descriptions copied verbatim).
- The **same daily firing time** and the **same number of fires per topic per substrate** (recommended: 7 fires per topic per substrate over the 7 days).

What necessarily differs: kagent runs each topic in its own pod under its own JOSE capability bundle; orchestrator runs all topics in one CronJob process. That's the substrate difference being measured. Don't try to neutralize it.

## 4. What to measure

Every metric below has: name, formula, data source on each substrate, storage schema, and a CSV column. The user produces one CSV per substrate per day (`evidence/v01-comparison/{kagent,orchestrator}-YYYYMMDD.csv`), and one merged summary CSV at the end of the week.

### 4.1 Wall-clock time per topic to first useful artifact

- **Metric:** `wall_clock_seconds = first_artifact_at - run_started_at`
- **Definition of "first useful artifact":** for kagent, `AgentTask.status.artifacts[0]` populated OR `status.result.content` non-empty (whichever happens first); for orchestrator, the digest markdown's first non-frontmatter byte hitting the PVC.
- **Data source:**
  - kagent: `AgentTask.status.startedAt` and `status.completedAt` (both already shipped) — also crosschecked against the OTel root span's `start_time` / `end_time` via Langfuse query.
  - orchestrator: JSONL trace's `runStartedAt` / `runCompletedAt` envelopes (already in `agent-runtime`'s sink).
- **Storage:** CSV column `wall_clock_seconds` (float, 3 decimal places).

### 4.2 Number of LLM calls to converge

- **Metric:** `llm_call_count = count(iterations where llm.request was non-empty)`
- **Data source:**
  - kagent: count of `gen_ai.operation.name=chat` spans under the run-root span in Langfuse.
  - orchestrator: count of `iteration` envelopes in the JSONL trace where `llmRequest != null`.
- **Storage:** CSV column `llm_calls` (int).

### 4.3 Token cost per topic

- **Metric:** `cost_usd = sum(input_tokens * input_price) + sum(output_tokens * output_price)` per run.
- **Data source:**
  - kagent: `langfuse.observation.cost_details.total` summed across the run's spans (already populated by `OtelTraceSink` via the `gen_ai.usage.{input,output}_tokens` attributes — `0e26ad1` and the Langfuse semconv reshape).
  - orchestrator: `RunBudget.cumulativeCostUsd` from the result envelope (already computed in-loop by `agent-runtime`).
- **Storage:** CSV columns `input_tokens` (int), `output_tokens` (int), `cost_usd` (float, 4 decimal places).
- **Pricing source:** read from `Agent.spec.llmParams.pricingHint` if set; else use the Langfuse model-cost table; else use the gateway provider's published rate. Fix one source for the whole week — don't switch mid-run.

### 4.4 Number of failures, retries, restarts

- **Metric:** four counters, summed over the run:
  - `failures` = run ended with `phase=Failed` (kagent) or non-zero exit + non-empty error in JSONL (orchestrator)
  - `retries` = number of LLM calls that returned `5xx` / `429` / timeout and were retried (per the loop's retry middleware)
  - `pod_restarts` = `containerStatuses[].restartCount` at terminal (kagent only; orchestrator runs in a single process, so always 0 or 1=process crash)
  - `human_interventions` = manually counted (Section 5.3)
- **Data source:**
  - kagent: `AgentTask.status.phase`, `containerStatuses`, the audit stream (`task.failed`, `task.retried` per `v0.1.15-audit-stream`).
  - orchestrator: JSONL trace `runEnded` envelope's `error` field; the CronJob's pod termination state.
- **Storage:** CSV columns `failures`, `retries`, `pod_restarts`, `human_interventions` (all int).

### 4.5 Operator-side audit-event volume

- **kagent only.** Useful as observability ROI — homelab-orchestrator emits no audit stream.
- **Metric:** `audit_event_count = count(CloudEvents emitted on NATS subject pattern kagent.audit.* during the run window)`
- **Data source:** NATS JetStream consumer over `kagent.audit.>` filtered by `data.taskUid == <run uid>`.
- **Storage:** CSV column `audit_events` (int; null for orchestrator runs).
- **Note:** this is not a comparison axis. It is reported to demonstrate that observability cost rides at zero pod-cost for kagent (since the audit publish is fire-and-forget on JetStream) and to seed pricing conversations for any later commercial deployment.

### 4.6 "Got stuck" — human intervention required

- **Metric:** `human_interventions` (already in 4.4).
- **Definition:** any case where the run did not reach a terminal state on its own and required the operator to (a) `kubectl delete` a stuck Pod, (b) restart the CronJob, (c) bounce the operator deployment, (d) manually patch a status field, (e) clear a queue / topic, or (f) restart the cluster. Each of these counts as **+1** human intervention for that run.
- **Data source:** manual log kept in `evidence/v01-comparison/interventions.md` — append-only, format: `YYYY-MM-DD HH:MM substrate=<kagent|orch> topic=<slug> action=<a|b|c|d|e|f> reason=<short text>`.
- **Disqualification rule:** see Section 5.3.

### 4.7 Storage layout

```
evidence/v01-comparison/
├── kagent-YYYYMMDD.csv         # per-day, one row per topic
├── orchestrator-YYYYMMDD.csv   # per-day, one row per topic
├── interventions.md            # append-only human-intervention log
├── runs/                       # raw evidence
│   ├── kagent/
│   │   └── <task-uid>/
│   │       ├── status.yaml     # kubectl get agenttask <name> -o yaml at terminal
│   │       ├── trace.json      # Langfuse trace export
│   │       └── audit.jsonl     # NATS audit events filtered to this taskUid
│   └── orchestrator/
│       └── <run-id>/
│           ├── digest.md       # the artifact
│           └── run.jsonl       # the full trace
└── summary.csv                 # merged: 7 days × 5 topics × 2 substrates = 70 rows
```

CSV header (both per-day and `summary.csv`):

## 5. One-week protocol

### 5.1 Day 0 — preconditions

Before day 1 (do all of these and check them off in `evidence/v01-comparison/preconditions.md`):

- [ ] kagent at HEAD (post-rev2 fixes) deployed to homelab K3s via ArgoCD, smoke test green.
- [ ] homelab-orchestrator deployed to the same cluster in a separate namespace (`homelab-orchestrator`) with its CronJob suspended (`kubectl patch cronjob -n homelab-orchestrator orchestrator -p '{"spec":{"suspend":true}}'`).
- [ ] Same gateway URL + same model + same `llmParams` configured on both sides. Capture both configs:
      `kubectl get cm -n kagent-system kagent-operator-config -o yaml > evidence/v01-comparison/kagent-config.yaml`
      `kubectl get cm -n homelab-orchestrator orchestrator-config -o yaml > evidence/v01-comparison/orchestrator-config.yaml`
- [ ] Topic 4 (`k8s-operator-design-activity`) added to both substrates' topic lists (kagent: AgentTask manifest; orchestrator: edit `topics.yaml` + redeploy).
- [ ] Topic 5 (`paperclip-mode`) re-enabled on the orchestrator side (uncomment in `topics.yaml`); on kagent it ships as a separate AgentTask.
- [ ] Langfuse reachable from kagent agent-pods; verify with one trial AgentTask.
- [ ] PVC for orchestrator JSONL traces has ≥10GB free.
- [ ] `evidence/v01-comparison/` directory created in this repo on a branch (do NOT commit raw run data to main; squash to a single results file at the end).

### 5.2 Daily protocol (days 1-7)

Each day, both substrates fire each of the 5 topics once. Stagger them by an hour to avoid gateway-side rate-limit collision:

- 06:00 — orchestrator CronJob fires (single process, all 5 topics in sequence).
- 07:00 — Helm-charted CronJob (or scripted `kubectl apply`) creates 5 AgentTasks under kagent.
- 12:00 — daily check-in (Section 5.4).

The 06:00 / 07:00 split is intentional. If the gateway has a per-minute throttle and both substrates fire at the same time, the second fire collides with the first — making the comparison about gateway concurrency, not substrate quality.

For day-N AgentTask creation, use:

```bash
for slug in cloudflare-workers-ai mcp-spec-activity anthropic-platform-activity k8s-operator-design-activity paperclip-mode; do
  envsubst < evidence/v01-comparison/agenttask-template.yaml | kubectl apply -f -
done
```

The template uses `${SLUG}` and `${DATE}` env vars and renders an `AgentTask.metadata.name` of `comparison-${SLUG}-${DATE}` so each day's run is uniquely addressable.

### 5.3 Allowed and disqualifying interventions

**Allowed (count +1, run still valid):**
- a. Restart a cluster service that is not under test (NATS, ArgoCD, Langfuse). The agent-pod / orchestrator process under measurement was not touched.
- b. Re-create a stuck NetworkPolicy (orchestrator only — kagent does not yet ship NetworkPolicy).
- c. Restart the gateway pod (LiteLLM or whichever) for a non-substrate-related reason (homelab power blip, etc).
- d. `kubectl delete` an `AgentTask` that has been Pending for >30min with no logs, and re-fire it. **Counts as +1 human intervention AND +1 retry.**

**Disqualifying (this day's data for the affected substrate is dropped):**
- D1. Modifying the kagent operator binary, agent-pod image, chart values, or Agent CRs *during* the test week to fix a behavior that emerged.
- D2. Modifying `homelab-orchestrator`'s code, image, or `topics.yaml` *during* the test week.
- D3. Switching the gateway URL or the model mid-week.
- D4. Switching the LLM tier (e.g. promoting Scout → Sonnet) on either side.
- D5. Lengthening / shortening the topic source list mid-week.

If a disqualifying event happens, mark that day as `disqualified=true` in the day's CSV header line (added as a leading comment row), and the merged summary excludes it.

### 5.4 Daily check-in

12:00 each day, the operator runs:

```bash
./scripts/v01-comparison-daily-check.sh YYYYMMDD
```

(Script does not exist yet — adding it is part of the rig prep work; suggested ~150 lines that does: `kubectl get agenttask -n kagent-comparison -o yaml`, dumps Langfuse traces via API, snapshots audit-stream subset, queries orchestrator pod logs, writes the day's CSV row.)

The check-in rolls up that day's data into the day's CSV and appends to `interventions.md` if any happened. **Do not look at the CSV diff between substrates during the week** — that biases mid-week intervention. Look only at the end.

## 7. Pass / fail criteria

The comparison rig **passes** (kagent v0.1 ships) iff **all** of:

- **P1.** Median per-topic `wall_clock_seconds` does not regress by more than **20%** vs orchestrator baseline. Computed over the 35 (7 days × 5 topics) per-substrate samples; per-topic medians compared.
- **P2.** Median per-topic `cost_usd` does not regress by more than **50%** vs orchestrator baseline. (Allow more headroom on cost than on latency because kagent's per-pod cold-start adds a fixed-cost floor that orchestrator's in-process loop avoids.)
- **P3.** Per-topic completion rate (`% rows where phase=Completed`) is **≥ orchestrator's completion rate − 5 percentage points**. Strict floor: kagent must complete ≥ 90% of topics in absolute terms regardless of orchestrator's number.
- **P4.** `human_interventions` total over the week ≤ orchestrator's `human_interventions` total. Equality is fine — substrate-of-the-month should not be more brittle.
- **P5.** No disqualified day on the kagent side. (Disqualifications on the orchestrator side don't block — they make orchestrator look better, and we still have to beat that.)

The rig **fails** if any of P1-P5 is violated. A fail does **not** mean the project is dead — it means Wave 1 (Workspace + typed I/O + CAS) becomes the validation bet, per `WHY.md §6`. Wave 1 fails to move these same numbers → admit-failure territory.

The rig is **inconclusive** (re-run needed) if:
- Fewer than 5 valid days per substrate after disqualifications.
- Gateway provider pricing changed mid-week and the replay-pricing exercise can't reconstruct true cost.
- Fewer than 3 of the 5 topics completed at least once on each substrate.

In the inconclusive case, fix the cause and re-run a 7-day window. Do not patch the data.

## 9. Out of scope

- **Cluster-side execution.** Claude Code cannot run this. The user runs it on the homelab. Each day's check-in is human-driven.
- **kagent v0.2+ comparisons.** This document is the v0.1 ship-gate criterion specifically. Wave 1 (typed I/O + Workspace + CAS) is not measured by this rig — it has its own gate per `docs/SUBSTRATE-V1.md`.
- **Multi-tenant comparisons.** The homelab is single-user. Tenant-isolation behavior is `Wave 4` scope; this rig covers single-user shape only.
- **Untrusted-code-exec comparisons.** Both substrates run trusted research code; the Kata / Bubblewrap / Firecracker isolation story is `Wave 0+` and not exercised.
- **Cross-cloud comparisons.** kagent-on-EKS vs orchestrator-on-K3s is not this test. Same homelab, same K3s, both substrates side-by-side.
- **Framework-swap comparisons.** Strands TS vs Mastra vs forked AgentExecutor inside the same kagent pod is application-layer per `SUBSTRATE-V1.md §8` — out of scope.
