   - "kagent ships substrate-thin context-pressure handling — refusal at 95% + agent-managed handoff via `get_my_context.tokenUtilization` + `context_pressure_ignored` detector. No other OSS substrate ships this triple."

## 3. NEW finding: substrate context-pressure primitive

The user's question: *does ANY OSS competitor today ship a substrate-level context-pressure safety-net (intercepts the loop when ctx > 95%) plus a quality detector for "agent ignored its own context budget"? This may be a new defensible primitive — or it may already exist in autogen-core / langgraph runtime.*

**Answer: NO. kagent's three-piece slate is unique in the surveyed OSS landscape.** Detail below.

### 3.1 Comparison: how each peer handles approach to the model's context limit

| Project | Pre-call refusal at threshold? | Pod-side token-utilization introspection? | Quality flag for "agent ignored its budget"? | What it actually ships |
|---|---|---|---|---|
| **kagent** (this project) | ✅ **substrate refuses next call at 95% with `LLMClientHttpError(0, 'context_window_substrate_refused')`** (`73f67f4`) — distinct from upstream 400 | ✅ `get_my_context().tokenUtilization = {used, modelWindow, percentage}` (`fb549c0`) | ✅ `context_pressure_ignored` flag in `structuralVerdict.suspicious[]` when utilization > 70% AND no `spawn_child_task` in last N=3 iterations (`fc32b13`) | The "substrate-thin, agent-managed" pattern — substrate observes & refuses; application's prompt decides strategy |
| **Anthropic Claude Code** | ⚠ **auto-COMPACTION at ~95%** (different design) — substrate summarizes the conversation rather than refusing | ❌ no agent-side introspection tool | ❌ no quality flag for self-management failure | The "substrate-thick, auto-compact" pattern — substrate owns compaction quality. Per [GitHub issue #25679](https://github.com/anthropics/claude-code/issues/25679), threshold is hardcoded; users requesting configurability. **Different design philosophy.** |
| **autogen-core (microsoft/autogen, maintenance)** | ❌ no pre-call refusal | ⚠ `BufferedChatCompletionContext` truncates by buffer-size (most-recent-N messages); operates on context, not on token count | ❌ no quality flag | The "buffer-windowing" pattern — drops older messages when buffer fills. **Pre-tokenization; no token-aware behavior.** |
| **microsoft/agent-framework 1.0** | ❌ | ❌ documentation does not mention context-window awareness primitives ([repo README inspection](https://github.com/microsoft/agent-framework)) | ❌ | The "framework SDK" pattern — context handling deferred to provider SDKs |
| **langgraph 2.0** | ❌ no built-in pre-call refusal | ⚠ token-budget metadata can be attached via custom span metadata; not a runtime primitive | ❌ | The "checkpointer + observe" pattern — relies on LangSmith for monitoring, not on runtime intervention |
| **dapr/dapr-agents 1.0** | ❌ (verified via repo README inspection — no context-window safety mechanism mentioned) | ❌ | ❌ | The "durable workflow" pattern — focuses on resume-from-checkpoint, not on in-loop budget enforcement |
| **kubernetes-sigs/agent-sandbox** | ❌ | ❌ | ❌ | The "isolation primitive" pattern — explicitly out of scope for what the agent does inside the pod |
| **AgentField** | ❌ | ❌ | ❌ | The "identity + audit" pattern — context is application's problem |
| **Strands / Mastra / CrewAI / LangGraph.js / OpenAI Agents SDK** | ❌ in any surveyed | ❌ | ❌ | SDK-level pre-call hooks exist (e.g., `stopWhen` in Vercel AI SDK 6) but none enforce on `cumulative >= 0.95 * window` with both refusal + introspection + detector |

### 3.2 Why the kagent design is *substantively* different from Claude Code's auto-compact

Both ship "do something at 95%." But the design choices diverge sharply:

| Dimension | kagent (substrate-thin) | Claude Code (substrate-thick) |
|---|---|---|
| What the substrate does | Refuse the next call (terminal `Failed`) + emit detector flag | Summarize the conversation (auto-compact) and continue |
| Who owns compaction quality | Application's prompt (it must call `spawn_child_task` with a hand-written brief) | Substrate owns the summarizer model + summary template |
| Failure mode if substrate is wrong | Loud, observable, retry-able (`error: 'context_window_substrate_refused: ...'`) | Silent loss-of-information; downstream symptoms only |
| Composability with custom prompts | Application can inspect `tokenUtilization.percentage` and act differently per workload (researcher preserves URLs; coder preserves file paths; monitor preserves last-N events) | Application can't change the substrate's compaction policy |
| Per-prompt observability | `context_pressure_ignored` flag attributes "this prompt is bad at self-management" to a specific Agent CR | No such attribution; compaction is invisible to the operator |

**The argument in `docs/CONTEXT-AWARENESS.md` §2 is correct against the surveyed alternatives**: auto-compaction makes the substrate own a strategy decision (when to compact, what to keep) that is application-shaped. kagent's design refuses to make that decision.

### 3.3 The defensibility test

For a primitive to count as defensible, three conditions must hold:

1. **Real (not stylistic) novelty.** ✅ Verified above — no surveyed project ships the kagent triple.
2. **Earned by experimentation, not by spec-then-build.** ✅ The detector battery (F1/F2/F3 + refusal + synthesis-vacuity + now context_pressure_ignored) was lifted from the chat-server experiments documented in `docs/HARNESS-LESSONS.md`. This is *learned* substrate behavior, not invented.
3. **Belongs at the substrate layer.** ✅ The argument in `CONTEXT-AWARENESS.md` §2 is sound — auto-compaction belongs to applications; *budget enforcement + observability + flagging* belongs to the substrate. The slate ships exactly the substrate-shaped concerns and intentionally avoids the application-shaped ones.

## 5. Strategic recommendation

Given today's landscape, the single sharpest claim kagent can defend is a **two-clause** claim — both clauses evidence-backed:

> **"kagent is the only OSS K8s-native agent operator that ships (a) substrate-thin context-pressure handling — pre-call refusal + agent-side introspection + quality detector — and (b) sealed-JWT capability narrowing where the substrate enforces `child.claims ⊆ parent.claims` at the spawn boundary."**

Evidence for clause (a) — three commits in the last 30 days:
- `73f67f4 feat(agent-loop): substrate context-window safety-net at 95% threshold`
- `fb549c0 feat(agent-pod, agent-loop): plumb contextWindowTokens through RunBudget + get_my_context`
- `fc32b13 feat(agent-loop): context_pressure_ignored detector`

Plus the verified absence of equivalent designs in: kubernetes-sigs/agent-sandbox v0.4.5, kagent.dev v0.9.2, Kagenti v0.5.1, Dapr Agents v1.0.2, Microsoft Agent Framework 1.0, AgentField v0.1.77, autogen-core (maintenance), langgraph 2.0, Strands, Mastra, CrewAI, OpenAI Agents JS, Claude Agent SDK.

Evidence for clause (b):
- The prior audit's R1.4 #4 finding (no other surveyed OSS project ships caveat-narrowing JWT capabilities at the K8s spawn boundary) is **still valid as of 2026-05-06**: zero KEPs and zero open issues in `kubernetes-sigs/agent-sandbox` mention capability/identity/SPIFFE; AgentField's W3C-DID approach is **issuance-time identity, not narrowing-on-spawn enforcement**; Kagenti uses Keycloak RFC-8693 token exchange (also not narrowing); MAF uses Ed25519 plugin signing (not capability narrowing).

### 5.1 What this claim explicitly avoids

- Does NOT claim "only OSS K3s-native" — falsified.
- Does NOT claim novelty for per-pod isolation — agent-sandbox owns that primitive and is shipping faster.
- Does NOT claim novelty for SPIFFE identity — Kagenti's pattern is stronger.

This is the marketable distinction; the threshold itself is not.
