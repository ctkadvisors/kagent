# R3 (RE-RUN) — In-pod SDK landscape vs. context-awareness slate

**Date:** 2026-05-06
**Range under review:** `7032be9..fc32b13` (six commits introducing the v0.1.9 context-awareness slate)
**Method:** Read-only re-verification of the prior R3 against the current substrate. Each candidate SDK re-checked via web search May 2026; in-pod files cited at line level.
**Bottom-line shift vs. prior R3:** The "any framework runs in a pod" claim REGRESSES with the new safety-net. The substrate now silently drops a guarantee for any in-pod runtime that is not `@kagent/agent-loop` — third-party SDKs neither read `KAGENT_AGENT_MODEL_CONTEXT_WINDOW` nor honor the 95% refusal nor surface the `context_pressure_ignored` flag. The bar to call them "drop-in" rose by one substrate primitive.

---

## 0. What the slate actually changed (load-bearing context for the rest)

The four pieces in `docs/CONTEXT-AWARENESS.md` ship as a tightly-coupled bundle that is **fully internal to `@kagent/agent-loop` + the `@kagent/agent-pod` runner**:

1. **Operator-side env projection.** `KAGENT_AGENT_MODEL_CONTEXT_WINDOW`, `KAGENT_CONTEXT_SAFETY_THRESHOLD` (default `0.95`), `KAGENT_CONTEXT_PRESSURE_THRESHOLD` (default `0.7`) are stamped on every spawned pod. (`docs/CONTEXT-AWARENESS.md:88-91`.)
2. **Agent-pod env reader.** `parseEnv` parses the window into `PodConfig.contextWindowTokens` and threads it forward. (`packages/agent-pod/src/env.ts:361,379`.)
3. **Runner threading.** `runner.ts` reads the pod-level safety + pressure thresholds from env and threads `contextWindowTokens`, `contextSafetyThreshold` onto `executor.run()`. The pressure threshold is only consumed by the detector. (`packages/agent-pod/src/runner.ts:286-288, 304-307, 317-326`.)
4. **Executor pre-call refusal.** `chatWithRetry` checks `cumulativeInputTokens + cumulativeOutputTokens >= safetyThreshold * contextWindowTokens` BEFORE every `LLMClient.chat()` call and throws `LLMClientHttpError(0, 'context_window_substrate_refused: …')` if exceeded. (`packages/agent-loop/src/executor.ts:410-431`.)
5. **Detector.** `computeQualityFlags` now appends `context_pressure_ignored` when utilization crossed the pressure threshold AND `spawn_child_task` was not called in the last N iterations. (`packages/agent-loop/src/detectors/quality-flags.ts:85-101, 123-141`.)

Every one of these is gated on `RunBudget.contextWindowTokens !== undefined` — the entire chain is a **no-op when the model-class chart entry omits `contextWindowTokens`** (back-compat by design, per `docs/CONTEXT-AWARENESS.md:93-101`).

The critical observation: **the safety-net is INSIDE `@kagent/agent-loop`'s `chatWithRetry`**. The agent-pod runner owns the env→budget plumbing but does NOT enforce or audit the contract from outside the executor. The detector runs in `runner.ts` AFTER `executor.run()` returns, against `result.budget` — which is populated by the executor from its own internal accounting. Both load-bearing safety surfaces live in the same package an alternative SDK would replace.

## 2. Context-pressure compatibility matrix

The question is bilateral: (a) does the SDK honor the substrate's 95% refusal cleanly when `LLMClientHttpError(0, 'context_window_substrate_refused: …')` is thrown by the model layer? (b) does the SDK expose the cumulative-token-utilization callback the substrate would need to surface a `context_pressure_ignored`-equivalent signal?

| SDK | (a) Will gracefully terminate on substrate refusal? | (b) Exposes per-step token usage to caller? | (c) Exposes cumulative-utilization callback? | (d) Has its own context-aware loop that COMPETES with substrate? |
|---|---|---|---|---|
| **Vercel AI SDK** v6 | YES — any thrown error from the wrapped model bubbles cleanly out of `streamText`/`generateText`. | YES — `onStepFinish({usage})` and `prepareStep({steps})` carry usage per step. | NO — caller must accumulate manually across `onStepFinish` invocations. | NO (as of v6.0). Caller-managed; middleware is the recommended pattern. |
| **Strands TS** 1.0 | PARTIAL — `SlidingWindowConversationManager` registers `AfterModelCallEvent` to "reduce context on overflow errors and request retry." The substrate's status=0 refusal would either be swallowed by retry-on-overflow OR propagate, depending on the manager's error matcher. UNVERIFIED. | YES — exposed on the AgentResult. | NO — caller hooks into AfterModelCallEvent and does the math. | YES — `SummarizingConversationManager` is auto-summarization, the exact pattern `docs/CONTEXT-AWARENESS.md §2` argues against. |
| **Mastra** | YES — provider errors propagate. | YES — exposed via memory layer. | PARTIAL — Observational Memory has its own observer agent that effectively does this, but wires off threshold per-thread, not per-substrate-call. | YES — Observational Memory is auto-compaction + auto-summarization. Strong overlap with what the kagent design DELIBERATELY does not ship. |
| **OpenAI Agents JS** | YES — model errors propagate. | PARTIAL — exposed on `RunResult.rawResponses`, not via per-step hook. | NO. | NO — only `maxTurns` cap. |
| **Claude Agent SDK** | NO — SDK auto-compacts before refusal could land. AND license-incompatible AND not LiteLLM-routable, so the question is moot. | N/A (subprocess). | N/A. | YES — auto-compact + memory tools + skill-progressive-disclosure. |
| **LangGraph.js** | YES — but only inside an `interrupt()` checkpoint pattern; default react agent will explode. Custom node required. | YES via state. | NO — caller must add a summarization node. | PARTIAL — community pattern is `trim_messages`/summary node. Caller writes it. |
| **Pydantic AI** | YES — `ModelHTTPError` propagates. | PARTIAL — `RunUsage` on result, not on `RunContext` (issue #4538 open). | NO — feature request #4137 is open. | NO native; `history_processors` are advisory. |
| **AutoGen v0.4** | YES — exceptions propagate. | YES via `model_client.create_stream` callbacks. | NO — caller must sum across messages. | PARTIAL — `BufferedChatCompletionContext` pre-truncates input. |
| **CrewAI** | PARTIAL — `respect_context_window=True` masks the refusal under auto-summary; `False` throws CrewAI's own exception (not wrapped). | PARTIAL via callbacks. | NO. | YES — `respect_context_window` IS auto-compaction. |
| **DSPy** | YES — exceptions propagate from LM. | NO not natively. | NO. | NO native. |

**Reading of the matrix:** Every SDK can be made to gracefully terminate on the substrate's refusal IF the adapter wraps the model client and lets `LLMClientHttpError(0, …)` propagate. None of them surface a substrate-equivalent `context_pressure_ignored` signal natively. Three of them (Strands TS, Mastra, CrewAI, Claude Agent SDK) ship their own auto-compaction that COMPETES with kagent's "deliberately not auto-compact" doctrine — meaning even if the substrate's threshold is right, the in-pod SDK summarizes silently before the threshold hits, defeating both the safety-net AND the detector's diagnostic value.

## 3. REGRESSION? — does the new safety-net + detector make the substrate LESS composable?

**Yes. Quantifiably.** Three regressions, in descending severity:

### 3.1 The "any framework in a pod" claim shifts from "true with caveats" to "true with FOUR caveats"

The prior R3.3 verdict (`docs/AUDIT-2026-05-06.md:191-201`) named three friction points: substrate tools, capability JWT, OTel semconv. The slate adds a fourth:

> **Friction 4 — Context-window safety-net.** Substrate sets `KAGENT_AGENT_MODEL_CONTEXT_WINDOW` and expects the in-pod loop to honor a 95% threshold. `@kagent/agent-loop` does this in `executor.ts:410-431`. NO third-party SDK reads this env var natively. A bare Vercel-AI-SDK / Strands-TS / Mastra agent-pod boots, ignores the env, and runs to upstream `400 context_length_exceeded` — which is exactly the failure mode the slate was designed to prevent. The "fail clean before the upstream's terminal 400 lands" guarantee is silently gone.

The shape of the regression is doubly bad: the substrate now ships an env contract that says "if you want safety, read this," but third-party in-pod runtimes cannot know to read it. The kagent docs call this contract out (`docs/CONTEXT-AWARENESS.md §3 Piece 1`), but no in-pod SDK's docs reference it.

### 3.2 The detector silently no-ops for non-`@kagent/agent-loop` runtimes

`computeQualityFlags` is invoked in `runner.ts:320-326` AFTER `executor.run()` returns, against `result.budget`. The budget is populated by `@kagent/agent-loop` from its OWN per-call accounting in `executor.ts:774-782`. A third-party SDK plugged in via a `RunDeps.toolProviders`-shaped adapter would not naturally produce a `RunResult.budget` shape that the detector reads — meaning:

- `context_pressure_ignored` flag does NOT fire for third-party in-pod runtimes.
- The diagnostic value of "operators learn which agent prompts are bad at self-management" (`docs/CONTEXT-AWARENESS.md §4.6`) is silently lost for any agent that doesn't use kagent's executor.
- An operator looking at `status.structuralVerdict.suspicious[]` cannot distinguish "agent didn't trip the detector" from "agent-pod ran a non-kagent loop that doesn't surface budget."

### 3.3 The substrate does NOT trip the detector or fail the run if the in-pod runtime ignores the env

I explicitly looked for this enforcement at the substrate boundary. **There is none.** The runner threads `contextWindowTokens` into `executor.run()` (`runner.ts:304-307`), and the executor enforces the threshold internally (`executor.ts:410-431`). The agent-pod's `main.ts` does NOT inspect outbound LLM traffic, does NOT mediate the `LLMClient` connection at the network boundary, and does NOT install a sidecar to sniff cumulative token usage. A third-party SDK that wires its own `OpenAICompatibleLLMClient`-equivalent against `KAGENT_LITELLM_BASE_URL` and never reads `KAGENT_AGENT_MODEL_CONTEXT_WINDOW` will:

1. Run until upstream `400 context_length_exceeded` lands.
2. Status patcher writes `phase: 'Failed'` with the upstream's terminal error message (per `docs/GATEWAY-CONTRACT.md §10`).
3. The `context_pressure_ignored` detector is NOT invoked because the run never returned a `RunResult` with the kagent-shaped `budget`.
4. The operator sees a generic upstream-400 failure, NOT the structured `context_window_substrate_refused:` reason the slate advertises.

Net: **the safety-net is opt-in for the in-pod runtime.** Third-party SDKs are not opted in.

### 3.4 Where the regression isn't terrible

- **Back-compat is preserved.** Existing agent-pods using `@kagent/agent-loop` work unchanged. Classes without `contextWindowTokens` in the chart degrade to no-op cleanly.
- **The substrate doesn't actively interfere.** A non-kagent runtime in a pod is not penalized for ignoring the env — it simply doesn't get the new guarantee.
- **An adapter is feasible per-framework.** ~80 LOC of `wrapLanguageModel` middleware would close the gap for Vercel AI SDK (see §4).
- **The detector failure mode is silent observation, not enforcement.** Missing `context_pressure_ignored` flags don't fail runs; they just degrade diagnostics.

## 4. What a `@kagent/agent-loop-vercel-ai` reference adapter would need

A reference adapter is the cleanest way to demonstrate that the "any framework in a pod" claim survives the slate. Concretely:

### 4.1 LOC budget

| Component | Estimated LOC | Function |
|---|---|---|
| `KagentContextSafetyMiddleware` (wrapLanguageModel) | ~80 | Read `KAGENT_AGENT_MODEL_CONTEXT_WINDOW` + `KAGENT_CONTEXT_SAFETY_THRESHOLD`. Maintain cumulative input/output token count across `doGenerate`/`doStream` calls. Throw before forwarding to underlying provider when threshold reached. Error class shape mirrors `LLMClientHttpError(0, 'context_window_substrate_refused: …')`. |
| `KagentSubstrateToolsAdapter` (re-emits `spawn_child_task`/`get_my_context`/etc. in Vercel AI SDK `tool()` shape) | ~120 | Per-tool wrappers that call into the existing kagent in-pod tool implementations. Same per-tool capability-claim cross-check as `runner.ts:resolveToolProviders`. |
| `KagentRunBudgetExtractor` (post-run, builds the kagent-shaped `RunBudget` so `computeQualityFlags` works) | ~40 | Walks `streamText` result steps → builds `cumulativeInputTokens`, `cumulativeOutputTokens`, `contextWindowTokens`. |
| `KagentTraceSinkAdapter` (Vercel AI SDK `experimental_telemetry` → kagent `TraceSink` shape) | ~60 | Maps AI SDK spans to kagent `TraceEntry` shape so detector callsite still receives compatible traces. |
| `runVercelAiAgentTask` (analog to `runAgentTask` in `runner.ts`) | ~80 | Boot path: parses `PodConfig` (reuses kagent env-reader), wires the four pieces above into a `streamText` call, returns `RunResult`. |
| Capability JWT enforcement wrapper (per-tool `execute` wrapper that consults the cap-bundle before invoking) | ~50 | Same pattern as `definePublishEvent`'s `policy_denied:no_capability` shape. |

**Total: ~430 LOC** for a fully-faithful adapter (preserves all four substrate guarantees: context-safety, capability JWT, substrate tools, context-pressure detector). Prior R3 estimate (`~100 LOC`) covered ONLY the bare loop boundary — that estimate was correct for "agent runs to completion" but did NOT cover any of the substrate guarantees.

If we accept "drop-in means the loop runs but loses substrate guarantees," the prior 100 LOC stands. If we accept the kagent doctrine "the in-pod runtime MUST honor the substrate's safety + cap + observability promises," the real number is ~430 LOC per framework.

### 4.2 Hook list (what the adapter MUST provide for each framework)

For `@kagent/agent-loop-vercel-ai`:

1. **Pre-call refusal hook.** `wrapLanguageModel` middleware reading the kagent context env vars and throwing the kagent-shaped error before forwarding to the underlying provider.
2. **Cumulative token accumulator.** `onStepFinish({usage})` sums `inputTokens + outputTokens` into a closure variable visible to (1).
3. **Tool-execute capability gate.** Per-`tool({execute})` wrapper that consults `loadCapabilityOptional` result (re-using kagent's `cap-consumer`).
4. **Substrate-tool re-emitter.** `tool()`-shaped wrappers around kagent's existing in-process tool implementations.
5. **`RunBudget` extractor.** Post-run `streamText` result walker that builds the budget shape the detector reads.
6. **`TraceSink` bridge.** Either via `experimental_telemetry: { isEnabled: true }` + an OTel SDK that re-emits to the kagent sinks, OR a manual `onChunk` capture that builds `TraceEntry` records.

For `@kagent/agent-loop-strands`:
- Items (1), (3), (4), (5), (6) the same.
- Item (2) replaced by Strands' `AfterModelCallEvent` hook, which already exposes per-call usage on the event payload.
- ADDITIONAL: a deliberate decision to disable `SummarizingConversationManager` so the substrate's safety-net actually fires (otherwise Strands' own auto-summary masks the threshold).

For `@kagent/agent-loop-mastra`:
- All six items required.
- ADDITIONAL: explicit configuration that Observational Memory's observer/reflector are EITHER off (so substrate's contract is the single source of truth) OR aligned to the same threshold (so they don't compete).

## 5. Verdict on "any framework runs in a pod"

**TRUE-WITH-CAVEATS, with one new caveat past the prior R3 verdict, and a sharpened LOC estimate.**

The headline shift: prior R3 estimated ~100 LOC of glue + ~200 LOC × N for substrate tools. Re-run R3 estimates ~430 LOC per framework if you want the in-pod runtime to honor the four substrate guarantees (context-safety, capability JWT, substrate tools, context-pressure detector). The slate added one new guarantee that no third-party SDK satisfies natively, raising the per-framework adapter cost by ~30%.

**The honest framing for the README:**

> kagent's substrate ships per-pod isolation, A2A messaging, observability, and (as of v0.1.9) a context-window safety-net. Any framework can run in a kagent pod, but if you want the safety-net + capability gating + the `context_pressure_ignored` diagnostic, you need either `@kagent/agent-loop` (the reference in-pod runtime) or a per-framework adapter that bridges the framework's hooks to the substrate's contracts. The substrate does NOT enforce the safety-net at the network boundary — third-party in-pod runtimes that ignore the `KAGENT_AGENT_MODEL_CONTEXT_WINDOW` env will run to upstream `400 context_length_exceeded`.

**Three concrete recommendations to close the regression** (in priority order):

1. **Ship `@kagent/agent-loop-vercel-ai` as the reference adapter.** Vercel AI SDK has the cleanest `wrapLanguageModel` middleware shape for this; it's the demonstrably-feasible path. ~430 LOC per the §4 budget. This makes the "any framework" claim provable rather than aspirational.
2. **Add a substrate-side enforcement option.** A future v0.2 ENVoy / sidecar / NetworkPolicy-mediated egress proxy could enforce the threshold at the wire boundary — making the safety-net work for ALL in-pod runtimes regardless of whether they read the env. Currently `egress-controller` package exists but does not parse LLM traffic. (`packages/egress-controller/`.)
3. **Document the regression honestly in `docs/DROP-IN.md` §8.** The current §8 says "no agent SDK — agents in pods run any framework." Add: "agents that don't use `@kagent/agent-loop` lose the v0.1.9+ context-safety guarantees unless they install a per-framework adapter; see `docs/CONTEXT-AWARENESS.md` §11 (TBD)."

## 6. Top 5 surprises (re-run)

1. **The safety-net is INSIDE the executor, not outside it.** I expected to find a `parseEnv`-time wire-mediator or an HTTP-client interceptor at the agent-pod's outbound LLM boundary. There isn't one. The substrate's "circuit breaker" lives in `chatWithRetry` (`packages/agent-loop/src/executor.ts:410-431`) — replacing the executor replaces the safety-net. (Verified by full read of `executor.ts` and `runner.ts`.)
2. **The detector is doubly substrate-coupled.** Not only does it read `RunBudget.contextWindowTokens` (from the executor), but its lookback walker reads `iteration_boundary` trace types (`quality-flags.ts:155-160`) — a kagent-`TraceEntry`-specific schema. A third-party SDK's traces would never produce this shape, so even if its budget were extracted, the lookback walker would fail-open ("no boundaries found, treat entire trace as one iteration") and over-fire the flag. (`quality-flags.ts:163-169`.)
3. **Mastra's Observational Memory (Feb 2026) is the ONLY framework that ships with kagent's exact problem framing built-in** — and it solves it the OPPOSITE way from the kagent doctrine. OM uses two background agents (Observer, Reflector) to compress old messages; `docs/CONTEXT-AWARENESS.md §2` explicitly argues against substrate-owned compaction for the same reasons OM is structured AROUND. The two designs are philosophically incompatible despite naming the same problem.
4. **Vercel AI SDK v6 finally has the hooks needed for this** — `prepareStep` + `onStepFinish` + `wrapLanguageModel` middleware compose cleanly into a `KagentContextSafetyMiddleware`. The community has already written drop-in middleware for context-window management (`https://community.vercel.com/t/.../38039`), confirming the pattern is mainstream. Six months ago this would have required forking the SDK; today it's middleware.
5. **The "any framework runs" claim was already overstated; the slate makes the gap legible.** Prior R3.3 was diplomatic about the friction. The slate exposes the gap concretely: there's now a substrate guarantee that demonstrably does not survive replacing the in-pod runtime, with no enforcement at the substrate boundary. This is good — explicit gaps are fixable; implicit gaps are not.
