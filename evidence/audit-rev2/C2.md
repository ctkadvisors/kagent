## 2. New BLOCKERs — agent-pod / agent-loop scope

### NB1 — `tokenUtilizationSnapshot` NOT wired in production; `get_my_context` always returns `used=0` [BLOCKER]

**Confidence: 100**

**Evidence:**

`packages/agent-pod/src/main.ts:359-363`:
```ts
const ctxDef = defineGetMyContext({
  podConfig: config,
  ...(capabilityBundle !== undefined && { capabilityBundle }),
  ...(remainingBudgetSeconds !== undefined && { remainingBudgetSeconds }),
});
```

The `tokenUtilizationSnapshot` dependency is not present. This dep is what wires live `RunBudget` state to the `get_my_context` handler.

`packages/agent-pod/src/builtin-tools.ts:1125`:
```ts
const snapshot = deps.tokenUtilizationSnapshot?.() ?? { used: 0, modelWindow: null };
```

Because the dep is `undefined` in production, the fallback fires unconditionally. The LLM always reads:
```json
{ "used": 0, "modelWindow": null, "percentage": null }
```

**Impact:** The entire CONTEXT-AWARENESS.md §2 design rationale states: "operators delegate to subagents with hand-written briefs because they don't trust the runtime to summarize itself — that discipline is what we want to support." The support mechanism is `get_my_context` returning an accurate `percentage`. In production, that percentage is always `null` (since `modelWindow` is also `null` due to `used=0` fallback cascading). An agent that calls `get_my_context` to decide "should I hand off now?" gets no actionable signal. The feature shipped in commits `fb549c0` and `fc32b13` but the critical wiring step was omitted from `main.ts`.

**Tests:** `builtin-tools.test.ts:981-1046` inject `tokenUtilizationSnapshot` directly into `defineGetMyContext`, so all tests pass while the production path is unwired.

**Fix:** In `main.ts`, after the executor run resolves OR by threading a shared `RunBudget` ref through the deps, wire `tokenUtilizationSnapshot` to return `{ used: budget.cumulativeInputTokens + budget.cumulativeOutputTokens, modelWindow: config.contextWindowTokens ?? null }`. Since the budget mutates in place during the run, a thunk reading it at call time is the correct pattern:

```ts
const ctxDef = defineGetMyContext({
  podConfig: config,
  ...(capabilityBundle !== undefined && { capabilityBundle }),
  ...(remainingBudgetSeconds !== undefined && { remainingBudgetSeconds }),
  // Wire the live budget ref so the LLM gets accurate utilization:
  tokenUtilizationSnapshot: () => ({
    used: runBudgetRef.cumulativeInputTokens + runBudgetRef.cumulativeOutputTokens,
    modelWindow: config.contextWindowTokens ?? null,
  }),
});
```

This requires either a shared mutable budget ref (matching how the executor already mutates `budget` in place) or a restructuring where `get_my_context` is registered after the executor is constructed and can close over its budget.

## 3. New HIGHs — agent-pod / agent-loop scope

### NH1 — `budget.tokensRemaining` reports the cap limit, not remaining capacity [HIGH]

**Confidence: 95**

**Evidence:**

`packages/agent-pod/src/builtin-tools.ts:1107-1113`:
```ts
const tokenLimit = podConfig.taskSpec.runConfig?.tokenLimit;
const budget: { tokensRemaining?: number; secondsRemaining?: number } = {};
if (typeof tokenLimit === 'number' && tokenLimit > 0) {
  budget.tokensRemaining = tokenLimit;
}
```

`tokenLimit` is the cap (e.g., 50000). The handler sets `tokensRemaining = 50000` without subtracting actual consumption. An agent that has consumed 49500 tokens and calls `get_my_context` reads `tokensRemaining: 50000`, not `tokensRemaining: 500`.

**Impact:** The `budget` field on `get_my_context` output is the actionable signal agents use for "should I spawn a child now?" (per CONTEXT-AWARENESS.md §4.4: "The agent's prompt is responsible for reading `percentage` and acting"). With `tokensRemaining` always reporting the ceiling rather than the remaining capacity, any prompt logic like "if tokensRemaining < 5000, hand off" never triggers. This compounds NB1: even if `percentage` were fixed, `tokensRemaining` would still give wrong guidance.

**Fix:** Subtract cumulative consumption from the limit. This requires the same `RunBudget` ref fix as NB1, or alternatively read from a snapshot dep:

```ts
const used = deps.tokenUtilizationSnapshot?.().used ?? 0;
if (typeof tokenLimit === 'number' && tokenLimit > 0) {
  budget.tokensRemaining = Math.max(0, tokenLimit - used);
}
```

### NH2 — H9 sub-issue: `Retry-After` has no upper-bound cap; sleep is not abort-interruptible [HIGH]

**Confidence: 85**

Prior H9 is PARTIALLY FIXED. Two distinct sub-issues remain:

**Sub-issue A — No Retry-After cap:**
`packages/agent-loop/src/executor.ts:343-344`:
```ts
if (retryAfterSec !== undefined) {
  return Math.max(0, Math.floor(retryAfterSec * 1000));
}
```
No `Math.min(...)` upper bound. A misbehaving or adversarial gateway returning `Retry-After: 3600` will cause a 1-hour sleep.

**Sub-issue B — Sleep is not abort-interruptible:**
`packages/agent-loop/src/executor.ts:479-480`:
```ts
await this.sleep(backoffMs);
if (llmCtx.abortSignal.aborted) { ... }
```
The abort check happens after `sleep` resolves. The default sleep implementation at `executor.ts:324` is `new Promise((r) => setTimeout(r, ms))`. There is no `AbortSignal` integration into the sleep itself. A `Retry-After: 30` sleep with SIGTERM at second 5 waits the full 30 seconds before the check at line 480 triggers cancellation. During that window, the kubelet's `terminationGracePeriodSeconds` may expire and SIGKILLs the pod before any status patch lands.

**Impact:** 429 storm from CF AI Gateway + large Retry-After = pod alive but useless for the sleep duration. Interaction with the safety-net: if the pod accumulated 94% context usage, receives a 429 during the next call, sleeps 10 minutes on Retry-After, and SIGTERM fires at minute 5 — the AgentTask fails with no `context_window_substrate_refused` reason and no `Failed` status patch because the SIGTERM handler can't unwind through the non-interruptible sleep.

**Fix:**
1. Cap Retry-After: `return Math.min(Math.max(0, Math.floor(retryAfterSec * 1000)), 30_000)` at `executor.ts:344`.
2. Make sleep abort-aware: replace `this.sleep` default with an abort-racing implementation, or add a `signal` parameter to the sleep dep and race `AbortSignal`.

## 4. New MEDIUMs / LOWs — agent-pod / agent-loop scope

### NM1 — `context_pressure_ignored` fires on runs that can't spawn (no escape hatch) — documented intent, but flagged as observable confusion [LOW/MEDIUM]

**Confidence: 75**

`packages/agent-loop/src/detectors/quality-flags.ts:119-121`:
> "the detector still fires when `spawn_child_task` is not in the agent's tool surface at all — that's a prompt-author bug worth flagging because the agent has no escape hatch."

This is per spec (CONTEXT-AWARENESS.md §4.6 last paragraph). However, in practice: for any Agent without `spawn_child_task` in its spec AND with `contextWindowTokens` set AND with a long run, `context_pressure_ignored` will fire on every completed task that exceeded 70% utilization. In a homelab with 10 long-running researcher agents that legitimately don't spawn children, this floods `status.structuralVerdict.suspicious[]` with a flag that the operator cannot tune away without setting `contextPressureThreshold > 1.0` (which fails the range validator and degrades to 0.7 default). There is no "I know this agent can't spawn, suppress the flag" escape. Consider a detector opt-out path via `Agent.spec` or a check for whether the tool is even admitted before firing.

### NM2 — `get_my_context` `budget.secondsRemaining` has a race vs. safety-net trip [LOW]

**Confidence: 80**

`packages/agent-pod/src/builtin-tools.ts:1108`:
```ts
const secondsRemaining = deps.remainingBudgetSeconds?.();
```

The `remainingBudgetSeconds` callback at `main.ts:313-316` is computed as `totalSec - elapsedSec` where `elapsedSec` is derived from the boot time vs. now. This is correct. However, the `get_my_context` handler captures `secondsRemaining` at call time, while the safety-net trip at `executor.ts:415` fires PRE-CALL. If the agent's last iteration saturated context, the safety-net fires before the LLM returns, before any tool call, so the agent never has a chance to call `get_my_context` to observe that it's near the edge. This is a by-design consequence of "safety-net fires pre-call," documented in CONTEXT-AWARENESS.md §4.5, but it means the agent's only window to introspect and hand off is during the tool-dispatch phase of prior iterations — not a bug, but operators should understand the LLM must call `get_my_context` INSIDE a tool-dispatch iteration, not as a standalone first message.

### NM3 — `tokenUtilizationSnapshot` dep missing also from the test runner's `resolveToolProviders` production path [MEDIUM]

**Confidence: 90**

The `resolveToolProviders` function at `runner.ts:597-651` and `runAgentTask` at `runner.ts:184-358` do not call `defineGetMyContext` at all — that happens in `main.ts` only. This means the `get_my_context` tool is NOT available when tests drive `runAgentTask` directly (e.g., integration-style tests of the full runner without going through `main.ts`). The tool is only wired inside `main.ts`'s `if (spawnEnabled)` block. If an operator writes a test using `runAgentTask` directly and lists `get_my_context` in `Agent.spec.tools`, the executor will throw "unknown tool" at boot. The tool should be universally wired by `resolveToolProviders` (or at minimum by `runAgentTask`) regardless of the spawn flag, since `get_my_context` is declared `UNIVERSAL` in the allowlist policy.

### NM4 — Tokenizer mismatch: executor uses `estimateTokens` fallback when backend doesn't report usage [MEDIUM]

**Confidence: 80**

`packages/agent-loop/src/executor.ts:774-778`:
```ts
budget.cumulativeInputTokens +=
  llmResult.usage?.inputTokens ??
  estimateTokens(currentMessages.map((m) => m.content).join('\n'));
budget.cumulativeOutputTokens +=
  llmResult.usage?.outputTokens ?? estimateTokens(llmResult.content);
```

When the backend (e.g., Cloudflare Workers AI) does not report `usage.inputTokens`, the executor falls back to `estimateTokens`, which is a character-count heuristic (`packages/agent-loop/src/trace.ts`). This estimate can be significantly off for models with different tokenization (Llama 4 vs. Claude vs. GPT-4). The safety-net's `used >= limit` check at `executor.ts:415` uses this potentially-inaccurate cumulative count. A model with aggressive tokenization could blow context without the safety-net firing; a model with conservative tokenization could trigger the safety-net prematurely. The design doc acknowledges "we trust the gateway's reported count" (CONTEXT-AWARENESS.md §8) but doesn't address the fallback path. When the gateway doesn't report tokens, the safety-net is operating on estimates that may be 20-40% off. Not critical (the 5% safety margin absorbs some of this), but operators should know the safety-net is approximate when the gateway doesn't report usage.

## 5. Net Delta — Security/Correctness Posture in Pod Scope

**Improvements since `32b5ff7`:**

- B1 FIXED: publish_event can no longer self-mint capability access. This was the highest-impact finding from the prior audit.
- B2 FIXED: fail-OPEN on missing JWT is now fail-LOUD by default with explicit opt-out. The operator now has a clear choice rather than a silent bypass.
- H7 FIXED: substrate tools (spawn, blackboard, events) now cross-check against Agent.spec.tools at boot. An agent cannot accidentally gain access to `spawn_child_task` without declaring it.

**New issues introduced by the context-awareness slate (`fb549c0`, `73f67f4`, `fc32b13`):**

- NB1 is a BLOCKER: The marquee feature of v0.1.9 — agent-managed context handling — does not work in production because `tokenUtilizationSnapshot` is unwired in `main.ts`. The LLM always reads `percentage: null`. Agents that implement context-handoff logic based on the `get_my_context` tool output will never trigger their handoff conditions. This will manifest as context blowups hitting the safety-net at 95% (when it IS wired, since `contextWindowTokens` flows correctly to the executor) but agents that attempted to hand off earlier will not see any pressure signal.

- NH1 is a HIGH: `tokensRemaining` in `get_my_context` reports the ceiling, not remaining capacity. Any agent prompt logic based on `budget.tokensRemaining` gives incorrect guidance.

- NH2 sub-issues (Retry-After cap + non-interruptible sleep) remain from the prior audit. The abort-check-after-sleep pattern is insufficient for protection against large Retry-After values.

**Remaining open BLOCKERs in C2 scope (from prior audit):** H11 (JWKS no retry/timeout — now partially mitigated by B2's catch, but the fetch itself is fragile), M6 (cap `['*']` bypasses allowedChildAgents).

**Overall posture:** The capability infrastructure is significantly stronger (B1, B2, H7 all fixed). The new context-awareness primitive is correctly designed at the protocol level but has a production wiring gap (NB1) that makes it a dead feature until fixed. The safety-net itself (executor.ts Piece 3) works correctly and is well-tested — it WILL refuse at 95% correctly when `contextWindowTokens` is set. The failure is in the observability/introspection layer (Piece 2 / `get_my_context`), not the enforcement layer.
