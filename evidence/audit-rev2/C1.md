## 3. New HIGHs

### NH1 — `KAGENT_CONTEXT_SAFETY_THRESHOLD: 0` in values.yaml silently loses the safety-net for all pods using `contextWindowTokens`

**Confidence:** 82

**File:line:** `packages/operator/charts/kagent-operator/values.yaml:243` (threshold value), `packages/operator/charts/kagent-operator/templates/deployment.yaml:429-430` (injection), `packages/agent-pod/src/runner.ts:904-908` (parse path)

**Evidence:** `parseContextSafetyThreshold` returns `undefined` when the value parses to `<= 0`. The executor then defaults to `0.95`. From the agent-pod operator's perspective this looks correct, but:

1. The chart has `contextSafetyThreshold: 0.95` default (safe).
2. If an operator sets `agentPod.contextSafetyThreshold: 0` to "disable" the safety-net, the chart stamps `KAGENT_CONTEXT_SAFETY_THRESHOLD=0` on the operator deployment. `buildJobSpecOptionsFromEnv` (main.ts:497-504) forwards it only when non-empty — `"0"` IS non-empty, so it is forwarded.
3. `parseContextSafetyThreshold("0")` → `n = 0 → n <= 0 → return undefined`. Runner receives `undefined`, executor defaults to `0.95`.
4. Result: operator who sets threshold to `0` to disable the net gets the opposite of what they expect (safety-net still fires at 95%). **No log, no error.** The misconfiguration is silent.

The symmetrical case (`contextSafetyThreshold: 1.1` → executor throws `InvalidConfigError` at run time, crashing the pod mid-run) is arguably worse, but the executor's validation at `executor.ts:524-533` does catch it for out-of-range values that survive the runner's parse.

**Fix:** Add a Helm `fail` guard in `_helpers.tpl` or `deployment.yaml`:
```
{{- if or (lt (float64 .Values.agentPod.contextSafetyThreshold) 0.01) (gt (float64 .Values.agentPod.contextSafetyThreshold) 1.0) }}
{{- fail "agentPod.contextSafetyThreshold must be in (0, 1]" }}
{{- end }}
```
And log a `WARN` in `parseContextSafetyThreshold` when the parsed value is rejected.

## 4. New MEDIUMs

### NM1 — `contextWindowTokens` for `source: 'override'` agents is permanently unset (silent safety-net bypass for pinned-model Agents in high-use clusters)

**Confidence:** 80

**File:line:** `packages/operator/src/model-class-resolver.ts:144-145`, `packages/operator/src/job-spec.ts:637-638`

**Evidence:** When `Agent.spec.model` is set (escape-hatch), `resolveAgentModel` returns `source: 'override'` with `contextWindowTokens: undefined`. `job-spec.ts:637-638` extracts `resolvedContextWindowTokens` only when `modelResolution.kind === 'resolved'` (always true for 'override') — but `contextWindowTokens` is `undefined` on the override arm. So `KAGENT_AGENT_MODEL_CONTEXT_WINDOW` is never stamped for override agents.

Per `CONTEXT-AWARENESS.md §9 Q5`, this is "intentionally excluded." The design decision is documented. However, the practical consequence is that a cluster that migrates most Agents to `modelClass` but leaves a few pinned (`spec.model: gpt-4o`) gets no safety-net for those Agents — silently. An operator who has enabled `contextWindowTokens` across their classes may assume the safety-net is cluster-wide, but it isn't for the pinned subset.

**Finding:** The design is correct per the doc; the doc does not explicitly warn that pinned-model Agents lose the safety-net. `MODEL-ROUTING.md §6` ("the escape-hatch") does not mention context-window implications.

**Fix (documentation):** Add a callout to `docs/CONTEXT-AWARENESS.md §7` and `docs/MODEL-ROUTING.md §6`: "Agents using literal `spec.model` (escape-hatch) do not receive `KAGENT_AGENT_MODEL_CONTEXT_WINDOW`. The safety-net and detector are no-ops for these pods. Migrate to `modelClass` to enable context-awareness."

### NM2 — Stale `contextWindowTokens` on in-flight Jobs after Helm upgrade (expected behavior, undocumented)

**Confidence:** 80

**File:line:** `packages/operator/src/main.ts:206` (`parseModelClassesEnv`), `packages/operator/src/job-spec.ts:637-695`

**Evidence:** `KAGENT_AGENT_MODEL_CONTEXT_WINDOW` is baked into each Job's pod env at spawn time. If a Helm upgrade changes `agent.modelClasses.tool-caller-default.contextWindowTokens` from `131072` to `200000`, in-flight pods running at `131072` will trip the 95% safety-net at `~124k tokens` instead of the new `~190k` ceiling. The reverse is also true: an upgrade that DECREASES the window produces pods that fire the safety-net later than the new threshold intended.

**Assessment:** This is correct-by-design (per `CONTEXT-AWARENESS.md §7`: "No agent-pod restart required for already-running tasks — they keep using the per-pod env from when they were dispatched"). However, neither `CONTEXT-AWARENESS.md` nor `MODEL-ROUTING.md` documents this explicitly as a caveat. An operator tuning the window down to tighten a runaway agent will find that already-dispatched pods are unaffected.

**Fix (documentation):** Add to `CONTEXT-AWARENESS.md §7`: "In-flight AgentTask pods are not affected by chart upgrades to `contextWindowTokens` — the value from the pod's spawn time is binding. Only newly-dispatched pods see the new value."

### NM3 — `KAGENT_CONTEXT_SAFETY_THRESHOLD` and `KAGENT_CONTEXT_PRESSURE_THRESHOLD` are forwarded verbatim (string) to all pods with no chart-level range enforcement

**Confidence:** 80

**File:line:** `packages/operator/charts/kagent-operator/templates/deployment.yaml:429-432`, `packages/agent-pod/src/runner.ts:904-908` (parseContextSafetyThreshold silently drops out-of-range), executor.ts:524-533 (throws for out-of-range values that pass through)

**Evidence:** The deployment.yaml stamps these env vars unconditionally as raw strings via `{{ .Values.agentPod.contextSafetyThreshold | quote }}`. No Helm-level validation. A chart value of `1.5` is forwarded as the string `"1.5"` → `parseContextSafetyThreshold("1.5")` → `n=1.5 > 1 → return undefined` → executor defaults to `0.95`. A value of `-0.5` → `n = -0.5 <= 0 → return undefined` → same silent default. This is the same silent-misconfiguration problem as NH1 but affects any out-of-range value.

For `contextPressureThreshold`, the runner reads it at runner.ts:318 via `parseContextPressureThreshold`. If out of range, the detector defaults to 0.7 silently.

**Fix:** Same Helm-level validation guard as NH1. Also add WARN logging in both parse helpers when a provided value is rejected so the misconfiguration appears in operator logs.

## 5. Pressure-Test Results

### PT1 — Stale modelClass after Helm upgrade
**Result: SAFE by design.** In-flight Jobs have env baked at spawn time. New dispatch picks up the upgraded chart's class map on the next operator reconcile (the operator pod is redeployed when its deployment env changes). No correctness problem. Documentation gap (NM2 above).

### PT2 — `contextWindowTokens: 0` or missing
**Result: GRACEFUL DEGRADATION.** `parseModelClassesEnv` drops `contextWindowTokens` with a warn-log when `<= 0` or non-integer (main.ts:243-255). The entry itself is kept, so the model class remains usable. `parseContextWindowTokens` in the agent-pod mirrors the same check (env.ts:446). All four pieces degrade to no-op when the field is absent. **No silent DoS, no stuck task.** This path is correctly implemented.

### PT3 — CRD schema for new fields
**Result: N/A.** `contextWindowTokens` is NOT a CRD field — it is a chart value on the operator's `values.yaml`, projected as an env var. No CRD migration required. The Agent CRD was NOT modified by these commits. OpenAPI validation concern does not apply.

### PT4 — Job budget (`activeDeadlineSeconds`) vs context-budget semantics
**Result: ORTHOGONAL, no conflict.** `activeDeadlineSeconds` comes from `task.spec.runConfig.timeoutSeconds` (job-spec.ts:900-907) — a wall-clock deadline. The safety-net fires on cumulative token count. They do not interfere: whichever limit is hit first terminates the pod cleanly. If the safety-net fires, the agent-pod writes `Failed` status and exits before K8s's deadline. If the K8s deadline fires first, the pod is SIGKILLed and the job-watch surfaces the failure (H8 race still exists here, but that's a pre-existing issue).

### PT5 — modelClass deletion mid-flight
**Result: CORRECT BEHAVIOR.** In-flight pods are unaffected (env already stamped). New AgentTasks for Agents using a deleted class → `resolveAgentModel` returns `unresolvable` → `job-spec.ts:624-627` throws → task gets `Failed` with `unresolvable_model_class: <class>` (loud, structured, per MODEL-ROUTING.md §3.1). No silent failure.

### PT6 — Double-injection / ARG_MAX pressure from new env vars
**Result: NO REGRESSION.** `KAGENT_AGENT_MODEL_CONTEXT_WINDOW` is a small integer string (e.g., `"131072"` — 6 chars). `KAGENT_CONTEXT_SAFETY_THRESHOLD` and `KAGENT_CONTEXT_PRESSURE_THRESHOLD` are small floats (e.g., `"0.95"`, `"0.7"`). Total new env payload is ~50 bytes. The pre-existing H12 concern (oversized `parentDistillation` in env-JSON fallback) is unaffected because the ConfigMap path is now the default (`useConfigMap: true`). The fallback path still lacks a floor cap but that's the pre-existing H12 finding, not a regression.
