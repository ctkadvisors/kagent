## 3. New HIGHs

### NEW-H1 — `contextWindowTokens` is injectable into pod env with no upper bound (Confidence: 82)

**File/Line**: `packages/operator/src/job-spec.ts:688-693` (contextWindowTokens projection); `packages/operator/src/model-class-resolver.ts:52-55` (ModelClassEntry shape)

**Description**: `contextWindowTokens` is read from the operator's `KAGENT_AGENT_MODEL_CLASSES_JSON` env, parsed by `parseModelClassesEnv` in `main.ts`, and projected verbatim onto spawned pods as `KAGENT_AGENT_MODEL_CONTEXT_WINDOW`. The `ModelClassEntry` interface at `model-class-resolver.ts:54` declares `contextWindowTokens?: number` with no maximum constraint. A cluster operator who sets `contextWindowTokens: 999999999999` in the Helm values causes the agent-loop's context-window arithmetic (`tokenUtilization.percentage = usedTokens / contextWindowTokens`) to always report near-zero utilization, permanently disabling the safety-net at 95% and the `context_pressure_ignored` detector. Misconfiguration vector, not a remote-code path, but the impact is silent: substrate's context-pressure protections all degrade to no-op cluster-wide.

**Fix**: Add validation in `parseModelClassesEnv` (operator `main.ts`) rejecting `contextWindowTokens` values outside `[1000, 2_097_152]` (covers GPT-3.5's 4K floor to hypothetical 2M ceilings). Log a structured error and skip the field rather than crashing the operator.

## 4. New MEDIUMs / LOWs

### NEW-M1 — PATCH /api/modelendpoints admits cluster-wide writes with no namespace scope check (Confidence: 85)

**File/Line**: `packages/workbench-api/src/routes/gateway.ts:312-388`; `packages/operator/charts/kagent-workbench/templates/clusterrole-actions.yaml:40-43`

**Description**: The PATCH route accepts arbitrary `namespace` and `name` path parameters and patches the named ModelEndpoint in that namespace via `patchNamespacedCustomObject`. The actions ClusterRole grants `modelendpoints: [patch, update]` cluster-wide. An authenticated workbench user can modify ModelEndpoint CRs in any namespace — including namespaces that belong to other tenants in a shared cluster. Combined with B6 (spoofable `X-Forwarded-User`), any in-cluster pod can tune model bounds in any namespace.

**Fix**: Restrict the actions ClusterRole for `modelendpoints` to the release namespace using a Role+RoleBinding instead of ClusterRole+ClusterRoleBinding. Enforce `namespace === defaultNamespace` check in `gateway.ts:312` until multi-namespace ModelEndpoint management is a declared requirement.

### NEW-M2 — /api/cluster/snapshot cross-contaminates allPods between two listNode calls (Confidence: 80)

**File/Line**: `packages/workbench-api/src/routes/cluster.ts:303-360`

**Description**: `cluster.ts:303` (`/api/cluster/snapshot`) reads `allPods = deps.cache.listPods()` at line 325 AFTER the async `listNode()` call at line 309. A pod lifecycle event between those two points causes the node pod-count to be computed against the post-event pod list while the node list is from pre-event. Cosmetic consistency bug under normal operation, but `listNode()` has no rate-limit and can be called arbitrarily many times per second by authenticated users — same as M14, now confirmed to also affect the `/api/cluster/nodes` endpoint (line 283).

**Fix**: Add a 5-second TTL cache around `listNode()` calls shared between both handlers.

### NEW-L1 — /api/gateway/capacity calls buildModelEndpointIndex on every request (Confidence: 80)

**File/Line**: `packages/workbench-api/src/routes/gateway.ts:253-260`

**Description**: Every `GET /api/gateway/capacity` call issues a `listClusterCustomObject` for all ModelEndpoints cluster-wide via `buildModelEndpointIndex`. There is no caching. Every Workbench page refresh (which triggers a capacity poll) issues an uncached cluster-wide list. In a cluster with many ModelEndpoint CRs this adds noticeable latency and apiserver load.

**Fix**: Cache the `buildModelEndpointIndex` result with a 5-second TTL, same as the M14 fix recommendation for node listing.
