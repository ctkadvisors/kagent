### H8 — JSON Patch test op for status writes: SUBSTANTIALLY CORRECT, ONE CORNER CASE

**Fix SHA:** `242181b`
**Verdict: SUBSTANTIALLY CORRECT. One low-probability corner case documented.**

`status.ts:236-275` — `writeStatus` iterates `['Dispatched', 'Pending']`, sending a JSON Patch `test` op on `/status/phase` for each. On 412, continues to next. On all 412s, drops silently.

**"status.phase is itself missing" concern from mandate:** RFC 6902 §4.6 defines `test` as "Tests that a value at the target location is equal to a specified value." If the path `/status/phase` does not exist (status never written), the test FAILS, returning 412. This means `writeStatus` would silently drop the patch for a brand-new AgentTask that never had `status` set.

In the normal K8s operator flow: the operator reconciler stamps `status.phase: Pending` on creation, then `Dispatched` on dispatch. The agent-pod boots ONLY after the Job is created. The sequence is always `Pending → Dispatched → (pod boots) → pod writes terminal`. So the `/status/phase` path EXISTS by the time any pod calls `writeStatus`. The corner case applies only if:

1. The operator crashes between creation and status-patching (leaving `status` absent), AND
2. The pod boots anyway (which requires the Job to exist, implying the reconciler got far enough to dispatch it).

This is a very low probability scenario. The consequence is the patch is silently dropped (the AgentTask stays in whatever incomplete state). Not a data corruption risk.

**Content-Type:** `jsonPatchOptions = setHeaderOptions('Content-Type', 'application/json-patch+json')` at line 58. Explicitly set. Correct.

**K8s status subresource test op:** The patch targets the `/status` subresource via `patchNamespacedCustomObjectStatus`. The apiserver evaluates JSON Patch ops against the full object representation of the status subresource (which IS the `/status` path). `test` op on `/status/phase` works correctly here.

**No new blocker. One acknowledged corner case (H8-CORNER) documented below.**

### H11 — JWKS retry: CONFIRMED FIXED

**Fix SHA:** `8dccd21`
**Verdict: CLEAN.**

`cap-consumer.ts:353-398` — `fetchJwksWithRetry`:
- `JWKS_FETCH_TIMEOUT_MS = 10_000` (not shown but referenced at line 362)
- `JWKS_RETRY_DELAYS_MS` — 3-attempt exponential-ish delays (referenced at line 363)
- Per-attempt `AbortController` with `setTimeout(() => controller.abort(), timeoutMs)` at line 368-369
- 3 attempts total (line 367: `attempt <= totalAttempts` where `totalAttempts = delays.length`)
- On terminal failure: throws `JwksUnreachableError`

**Startup probe interaction concern from mandate:** The JWKS retry with 3 attempts at (presumed) 250ms + 750ms + 2250ms delays = ~3.25s total worst case for the retry loop itself, plus 3 × 10s timeouts = up to 33s. K3s default startupProbe `periodSeconds=10` with `failureThreshold=3` gives 30s before CrashLoopBackoff classification. In the worst case (all 3 fetches time out at exactly 10s), the pod would exhaust its startup probe before retries complete, causing a CrashLoopBackoff classification even though the pod would have written a `Failed` status. However: (a) the retries complete long before the Pod container exits (they throw an error, which main.ts catches and patches); (b) the Pod exits with process.exit(1) after patching, which is a clean exit — kubelet sees successful container termination, not a crash. The startupProbe concern does not apply: `restartPolicy: Never` is set on the Job's pod template (`job-spec.ts` line visible in existing code). With `restartPolicy: Never`, CrashLoopBackoff never triggers. The pod fails cleanly once.

**No regression.**

### H12 — env-JSON 256 KiB cap: CONFIRMED MITIGATED

**Fix SHA:** `2dea7d6`
**Verdict: SUBSTANTIALLY MITIGATED.**

`env.ts:651`: `ENV_JSON_SPEC_PAYLOAD_MAX_BYTES = 262_144` (256 KiB).

**"Is 256 KiB the right number?" from mandate:** Linux `ARG_MAX` is typically 128 KiB but applies to the TOTAL exec environment (all env vars + args). The actual cap on a single env var's value under `execve` is lower. However:

1. The ConfigMap path is now the DEFAULT. `useConfigMap` defaults to true in `job-spec.ts`. The env-JSON path is the legacy fallback for one release.
2. `assertEnvJsonSpecBudget` sums ONLY `KAGENT_AGENT_SPEC + KAGENT_TASK_SPEC` byte lengths. It does not account for the other ~20 KAGENT_* vars the operator stamps. Total actual env block is larger.
3. 256 KiB for just the two spec vars is optimistic if the real headroom is ~80 KiB (128 KiB minus other vars). However, actual Agent.spec + Task.spec JSON is typically 1–10 KiB for realistic workloads. The 256 KiB ceiling would only be hit for pathological spec payloads (e.g., an agent with thousands of declared tools or a task with megabytes of payload). The Helm chart operator separately bounds these at the CRD level.

**Verdict:** The theoretical headroom calculation is pessimistic (the cap may be too high for extreme cases), but realistic production workloads are far below both 128 KiB and 256 KiB. Since the ConfigMap path is the default, the env-JSON cap is defense-in-depth for legacy deploys. Acceptable.

**No regression.**

## 3. New Findings — LOW Severity

### C2R3-LOW-1 — H8 + H11 interaction: JWKS boot failure status patch silently dropped for brand-new tasks

**Confidence: 80**

**Scenario:** JWKS retry exhausts at pod boot → `main.ts:183-207` calls `writeStatus(config, { phase: 'Failed', ... })`. If the AgentTask was created but the operator crashed before patching `status.phase: Pending`, then `/status/phase` does not exist. Both `test` ops (Dispatched, Pending) return 412. The `writeStatus` function drops silently per its H8 design. The AgentTask stays in an undefined phase with no terminal status.

**Probability:** Very low. Requires operator crash in the ~100ms window between AgentTask creation and `status: Pending` patching, PLUS the pod somehow booting anyway (which requires the Job to exist, implying the reconciler got past the dispatch step).

**Impact if hit:** AgentTask stuck with no terminal status. Operator reconciler may retry dispatch (creating another Job) or wait for human intervention depending on its state-machine guards.

**Fix path:** The H8 comment at `status.ts:38-43` documents the disjunction approach but notes "In practice the pre-terminal phase is almost always `Dispatched`." A more robust fix would test for `/status` object existence first, or use a conditional patch (`add` with `path: /status/phase` using RFC 6902's `add` semantics when the path doesn't exist). However, this is a low-probability edge case and the correct fix is on the operator side (ensure `status.phase` is always set before dispatching the Job). Filed as informational; no immediate code change required.

**File:line:** `packages/agent-pod/src/status.ts:184-185`

### C2R3-LOW-2 — `wrapGenerate` direct `.total` property access without optional chaining (vercel-ai adapter)

**Confidence: 82**

**File:line:** `packages/agent-loop-vercel-ai/src/context-safety-middleware.ts:172`

`result.usage.inputTokens.total` and `result.usage.outputTokens.total` are accessed without optional chaining. The `LanguageModelV3GenerateResult` type from `@ai-sdk/provider` defines `inputTokens` as a structured object (`{ total, noCache, cacheRead, cacheWrite }`). A provider that is non-conformant with the V3 spec OR returns a shape where `inputTokens` is undefined (possible for providers that haven't been updated to the V3 usage shape) would cause a `TypeError: Cannot read properties of undefined (reading 'total')` inside `wrapGenerate`.

The consequence: the middleware throws AFTER `doGenerate()` succeeds but before `recordUsage` is called. The exception propagates out of `wrapGenerate`. In `runVercelAiAgentTask`, this would be caught by the outer `try/catch` at `runner.ts:180-208` and classified as `status: 'failed'` (the catch-all `status = 'failed'` path at line 206-207). The safety-net `assertBelowThreshold` would NOT have been bypassed (it fired correctly before `doGenerate`), but the cumulative counter would NOT have been updated, meaning subsequent calls would undercount tokens. Combined with a conformant provider on the next attempt, the safety-net threshold check would use a stale (too-low) cumulative count — effectively weakening the safety-net for that run.

**Why not higher confidence:** The `LanguageModelV3` spec in AI SDK v6 requires the usage object shape. Production deployments using the AI SDK's built-in providers (OpenAI, Anthropic, Cloudflare) return the required shape. The risk is limited to custom or community providers that don't conform.

**Fix:** Replace line 172 with:
```ts
this.recordUsage(result.usage.inputTokens?.total, result.usage.outputTokens?.total);
```
`recordUsage` already guards `typeof input === 'number' && Number.isFinite(input)`, so optional chaining here passes `undefined` harmlessly to the existing defensive guard.

### C2R3-INFO-1 — `KAGENT_SPEC_SOURCE` annotation divergence: `'mixed'` case not stampable by operator

**Confidence: 85 (informational gap, not a runtime failure)**

**File:line:** `packages/operator/src/job-spec.ts:936-937` vs `packages/agent-pod/src/env.ts:358-359`

The operator stamps `kagent.knuteson.io/spec-source: configmap|env-json` on the Pod template annotation at Job creation time (operator knows which path it configured). The pod sets `process.env.KAGENT_SPEC_SOURCE = config.specSource` where `specSource` can be `'configmap' | 'env-json' | 'mixed'`.

The `'mixed'` case arises when `agentLoad.source !== taskLoad.source` — one file present, other absent, partial ConfigMap mount. The operator cannot predict this at job-create time (it cannot observe partial mounts). The annotation would say `configmap` while the pod's runtime WARN log and `process.env.KAGENT_SPEC_SOURCE` would say `mixed`. Observability tooling reading the annotation would see `configmap` while the actual runtime path was partial. The pod-side WARN log is correct; only the annotation is stale.

This is a monitoring gap, not a correctness issue. The pod still runs (resolving both specs from whatever source was available). Filed as informational per the M11 closure scope.

## 4. Shared Interface Stability — RunBudget and the vercel-ai Adapter

`RunBudget` is explicitly exported from `@kagent/agent-loop/src/index.ts:164-170` as a public API type:
```ts
export type {
  RunBudget,
  TerminalStatus,
  ExecutionResult,

```

The vercel-ai adapter imports it at `run-budget-extractor.ts:35`:
```ts
import type { RunBudget } from '@kagent/agent-loop';
```

`RunBudget` is a simple interface with five fields (`cumulativeInputTokens`, `cumulativeOutputTokens`, `cumulativeCostUsd`, `tokenLimit?`, `costLimitUsd?`, `contextWindowTokens?`). The vercel-ai adapter's `buildRunBudget` constructs a conformant `RunBudget` and passes it to `computeQualityFlags`. `computeQualityFlags` is also exported from `@kagent/agent-loop/src/index.ts:181-185`.

**Interface stability verdict:** STABLE. `RunBudget` is a public API export with no internal-only fields or unexported dependencies. The vercel-ai adapter would survive a refactoring of `AgentExecutor` internals as long as the `RunBudget` interface shape is preserved, which is enforced by the `export type { RunBudget }` surface contract. The adapter is NOT fragile to executor refactors.

The `computeQualityFlags` function signature (`traces, finalContent, userMessage, budget, opts`) is also public and stable. The adapter uses it correctly at `runner.ts:241-247`.

**One fragility noted (C2R3-LOW-2 above):** The `wrapGenerate` `.total` access is fragile to non-conformant providers, but this is within the adapter itself, not in the shared interface. The `RunBudget` interface itself is stable.
