# R3 (REV 3) — vercel-ai adapter as Potemkin proof: per-component fidelity audit

**Date:** 2026-05-07
**HEAD:** `f3f441d`
**Method:** Read-only adversarial audit of `@kagent/agent-loop-vercel-ai` (commit `2f2e484`) against the substrate guarantees that `@kagent/agent-loop` + `@kagent/agent-pod` actually deliver. Every claim cites file:line.
**Stance:** the adapter compiles, lints, types, has 36 tests passing, and meets coverage gates — every check the W4 report claims passes does pass. This audit pressure-tests whether those checks actually correspond to substrate-equivalent BEHAVIOR. They do not.

---

## 0. Bottom line

The adapter is a **partial Potemkin proof**: each component checks its surface-level shape in isolation, but the integrated runner has structural defects that prevent it from actually delivering substrate-equivalent behavior. The most damaging defect is a single-line oversight in `runner.ts:181-193` (no `stopWhen` threading) that reduces the entire loop to one step, regardless of `maxSteps` input. Test infrastructure mocks around all the integration paths that would catch this. The "any framework" claim does NOT survive the rev2 regression — it's now **TRUE-WITH-FIVE-CAVEATS**, one new caveat past rev2's four.

### Component 1 — `KagentContextSafetyMiddleware` — Score: **3 / 5**

**File:** `/Users/chrisknuteson/Projects/ctkadvisors/active/kagent/packages/agent-loop-vercel-ai/src/context-safety-middleware.ts`

**What it does well**

- Threshold check at `:138-146` mirrors `chatWithRetry`'s pre-call refusal at `executor.ts:547-568`. Same `cumulative >= safetyThreshold * window` math, same structured error message format.
- `KagentContextWindowRefusedError` carries `status = 0 as const` (`:55`) — matches the production `LLMClientHttpError(0, 'context_window_substrate_refused: …')` structure that prevents 429-retry from catching the refusal.
- Threshold validation at `:109-119` fails fast on out-of-range values, same shape as `executor.ts:651-660`.
- Stream wrapping via `TransformStream` at `:206-225` correctly observes terminal `finish` chunks for usage accounting.

**Material gaps**

1. **No `estimateTokens` fallback.** Per `docs/CONTEXT-AWARENESS.md` §8 NM6 the production agent-loop falls back to `estimateTokens(currentMessages.map(m => m.content).join('\n'))` when the upstream omits `usage.{inputTokens,outputTokens}` — see `executor.ts:959-963`. The middleware's `recordUsage` (`context-safety-middleware.ts:149-156`) only records when `Number.isFinite(input)`. AI SDK v3's `LanguageModelV3Usage.inputTokens.total` is typed `number | undefined` (verified at `node_modules/.pnpm/@ai-sdk+provider@3.0.10/.../language-model-v3-usage.ts:14`). When a provider returns `undefined`, the middleware silently records 0 tokens, the cumulative counter never advances, and the safety-net **never fires regardless of real utilization** — the exact failure mode CONTEXT-AWARENESS.md §1 was designed to prevent. Production agent-loop does NOT have this gap.
2. **Stream usage observation is async-lazy.** The `TransformStream` callback at `:208-223` only fires as chunks pass through, not synchronously after `doStream` resolves. If `streamText` begins the next step's `doStream` before the prior step's stream is fully drained, the cumulative counter for the next pre-call check is stale. Hard to demonstrate without running it, but the production executor accounting at `executor.ts:957-963` is synchronous (post-`llm.chat()` await), eliminating the race.
3. **Per-attempt re-check pattern not preserved.** `executor.ts:531-568` re-checks the threshold at the TOP of every retry attempt (within the for(;;) loop). The middleware checks once per `wrapGenerate`/`wrapStream` invocation; AI SDK retries inside `streamText` would not re-trigger the middleware's check.

### Component 2 — `KagentSubstrateToolsAdapter` — Score: **3 / 5**

**File:** `/Users/chrisknuteson/Projects/ctkadvisors/active/kagent/packages/agent-loop-vercel-ai/src/substrate-tools-adapter.ts`

**What it does well**

- Delegates to existing `InProcessToolDefinition` factories (`:175-193`) — substrate guardrails inside `defineSpawnChildTask` etc. are not re-implemented.
- `coerceArgs` at `:195-213` handles model-supplied stringified-JSON (a real Workers AI shape).
- AbortSignal forwarding at `:187` (with a defensive fallback `new AbortController().signal` that allocates per-call but never aborts — semantically correct, mildly wasteful).

**Material gaps**

1. **Per-tool capability binding is OPT-IN, not auto-wired.** `runner.ts:145-147` only passes `capabilityBindings` if the caller provides them. The adapter's outer cap wrapper (Component 6) only activates per binding entry; there are NO defaults. A consumer wiring `defineSpawnChildTask` who forgets the `spawn_child_task: { category: 'spawn', target: ... }` binding gets a tool that runs without the outer cap check — relying ENTIRELY on the inner factory's check (`builtin-tools-spawn.ts:259-277`). This is intentionally documented as "shape bridge, not policy bridge" (`substrate-tools-adapter.ts:30-31`), but it's a defense-in-depth regression vs. production where the agent-pod runner threads `parentCapability` and the inner factory's check fires unconditionally (`main.ts:435-440`).
2. **Permissive `inputSchema: z.unknown()`.** Documented at `:178-182` as a deliberate version-flexibility tradeoff. Result: the model sees no parameter schema for substrate tools — Vercel AI SDK's tool-call shaping cannot validate before sending. The kagent agent-loop's executor does enforce JSON-Schema validation (`executor.ts` builds `ToolDescriptor` from the JSON schema). Net: more model-side hallucinated tool calls land at the inner factory, which then refuses; observed behavior degrades from "tool works" to "tool errors and the model retries."
3. **The cap-narrowing on spawn re-emission does NOT flow through the wrapper.** `defineSpawnChildTask` narrows `parentCapability` based on `spec.capabilityClaims.spawn` for the CHILD's bundle (`builtin-tools-spawn.ts:124+`). The vercel-ai adapter's outer wrapper only checks the PARENT's cap matches the call. The child-cap narrowing is an inner-factory concern that fires correctly because the factory is passed through verbatim. Confirmed not bypassed.

### Component 4 — `KagentTraceSinkAdapter` — Score: **2 / 5**

**File:** `/Users/chrisknuteson/Projects/ctkadvisors/active/kagent/packages/agent-loop-vercel-ai/src/trace-sink-adapter.ts`

**What it does well**

- `iteration_boundary` markers are emitted (`:112-123`) — the lookback walker (`quality-flags.ts:182-210`) requires these.
- Monotonic sequence counter at `:109` — matches `executor.ts:24-27` invariant 1.
- Field shapes (llm_call at `:130-144`, tool_call at `:149-162`, run_complete at `:172-186`) align with the executor's emitted `TraceEntry` shapes.

**Material gaps**

1. **NOT WIRED TO A REAL TRACESINK.** The adapter is an in-memory array (`:108`) with a `traces()` accessor. Production traces flow to OTel + Langfuse + Stdout via `RunDeps.traceSinks` set in `main.ts`. Asked: "Where do traces go in production?" Answer: **Nowhere from this adapter.** The doc explicitly admits: `:32-39` "this is a structural bridge, not a full-fidelity `TraceSink`... integrators can pipe them into a real `TraceSink` after the run completes." A real Vercel-AI-SDK pod would lose all OTel + Langfuse observability unless the consumer wrote an additional layer to drain the array into actual sinks. This is a major regression vs. the production agent-pod runner where `OtelTraceSink` + `LangfuseTraceSink` are wired by default.
2. **`stringifyStepInput` is empty most of the time.** `:203-211` — falls back to empty string when `step.request?.body` is undefined (which is most steps). Result: `input_messages` carries `[{role:'user', content:''}]`, `input_tokens_est` falls back to `estimateTokens('')` = 0. Substrate observability degrades to "we know the model was called, not what it was sent."
3. **Boundary ordering mismatch.** The bridge opens a boundary BEFORE the first step (runner.ts:168) and AFTER each step (runner.ts:191). Production agent-loop emits boundaries at iteration starts. The detector's lookback walker treats boundaries as "delimiter" indices — the start of an iteration. The bridge's BEFORE+AFTER pattern produces N+1 boundaries for N steps. The walker logic at `quality-flags.ts:198-200` picks `boundaryIndices[max(0, len-N)]` as the window start; this still works because the walker's "last N iterations" semantics happen to be tolerant of double-boundary patterns, but it's not a perfect match — edge cases (1 step run, 0 step run) drift behavior subtly from the production invariants.

### Component 5 — `runVercelAiAgentTask` — Score: **2 / 5** (DEFECT BLOCKING PRODUCTION USE)

**File:** `/Users/chrisknuteson/Projects/ctkadvisors/active/kagent/packages/agent-loop-vercel-ai/src/runner.ts`

**Material defect — single-step loop**

The runner declares `maxSteps?: number` at `:92-93` with the comment `Mirrors AI SDK's stopWhen: stepCountIs(maxSteps)`. Then at `:181-193` it builds the streamText call:

```ts
const stream = (input._streamText ?? streamText)({
  model: wrappedModel,
  ...(input.systemPrompt !== undefined && { system: input.systemPrompt }),
  messages,
  tools: toolBundle.tools,
  ...(input.signal !== undefined && { abortSignal: input.signal }),
  onStepFinish: (step) => { ... },
});
```

**`maxSteps` is never threaded into the call.** Vercel AI SDK 6 defaults `stopWhen: stepCountIs(1)` (verified at `node_modules/.pnpm/ai@6.0.175_zod@4.3.6/.../stream-text.ts:309`). This means **the runner runs at most ONE LLM step regardless of caller input** — a tool-using agent loop where the model needs N round-trips terminates after the first one.

Existing tests don't catch this because they all use `_streamText` stubs that simulate multi-step behavior by manually calling `onStepFinish` multiple times (e.g. `runner.test.ts:128-181` synthesizes three steps via the stub callback). When a real `streamText` is wired, single-step termination is the actual behavior.

This is a fundamental defect that means the adapter, AS COMMITTED, would not run a real multi-iteration agent task. It would not be detected by any tests in the package.

**Other gaps**

1. **No env reader.** Production `runAgentTask` boots from `parseEnv()` (`agent-pod/src/env.ts`); the vercel-ai runner accepts pre-parsed values. A consumer must build their own env-parse layer to populate `contextWindowTokens`, `contextSafetyThreshold`, `contextPressureThreshold`, `runId`, `userMessage`. Documented obliquely in the README, but every consumer rebuilds 50+ LOC of env plumbing.
2. **No status writer.** Production `main.ts:518-560` writes phase/error to the Kubernetes AgentTask status. The vercel-ai runner returns a `VercelAiRunResult` shape that the consumer must serialize themselves. Substrate observability via `kubectl get agenttask` is a do-it-yourself integration.
3. **No SIGTERM handling.** Production `main.ts` wires SIGTERM to a clean shutdown path (per CLAUDE.md operational discipline). The vercel-ai runner accepts `signal?: AbortSignal` but doesn't connect it to process signals.
4. **No spawn-tool admission auto-detect.** Production runner derives `spawnToolAdmitted` from `agentSpec.tools.includes('spawn_child_task')` (`agent-pod/src/runner.ts:405-407`). The vercel-ai runner accepts `spawnToolAdmitted: boolean` from caller — if the caller forgets to thread it, defaults to `undefined` → `true`, firing the detector even on agents that don't admit spawn (the audit-rev2 NM4 regression the production runner specifically guards against).

## 3. Drop-in regression status

**Worsened.** Rev2 said "TRUE-WITH-FOUR-CAVEATS." The vercel-ai adapter was advertised as closing the rev2 regression. It does NOT — instead it adds a fifth caveat:

> **Friction 5 — The reference adapter is half-built.** The `@kagent/agent-loop-vercel-ai` package implements 6 components but the integrated `runVercelAiAgentTask` runner has a single-step loop defect (`runner.ts:181-193` — `maxSteps` declared but not threaded to `streamText`), no real `TraceSink` wiring (`trace-sink-adapter.ts:108` — in-memory array only), no env reader, no status writer, no JWKS verification of cap JWTs, and silently no-ops when the upstream omits `usage` (no `estimateTokens` fallback). The adapter compiles, lints, and has 36 passing tests, but the integration tests are stubbed in ways that hide all five integration gaps. A consumer following the README boots a Vercel-AI-SDK pod that runs ONE LLM step, drops all observability, and silently ignores the safety-net when the gateway omits usage.

The four pre-existing rev2 caveats remain — none of them is closed by the adapter:

1. ✗ Substrate tools — adapter is a SHAPE bridge, not a wiring bridge; consumer still wires the factories.
2. ✗ Capability JWT — adapter does not invoke `loadCapabilityOptional`; consumer still verifies the JWT.
3. ✗ OTel semconv — adapter does not emit OTel spans; consumer still wires the trace sink.
4. ✗ Context-window safety-net — middleware exists, but only fires when usage is reported AND the runner threads multiple steps (defective today).

## 4. Concrete gaps that will surface as production issues

In rough order of severity:

1. **Single-step loop (BLOCKER).** Any tool-using Vercel-AI-SDK agent deployed via this runner terminates after one LLM call. Symptom: agents that should iterate run once and return whatever the first call produced. Fix: thread `maxSteps` into `streamText({ stopWhen: stepCountIs(input.maxSteps ?? <default>) })` at `runner.ts:181-193`. One line.
2. **Silent-fail safety-net (HIGH).** When the gateway omits `usage.{inputTokens,outputTokens}.total`, cumulative counter stays 0, threshold never trips, agent runs to upstream `400 context_length_exceeded`. Symptom: substrate `context_window_substrate_refused` errors never appear; only generic upstream-400 failures. Production agent-loop has the `estimateTokens` fallback and does NOT have this gap. Fix: replicate `executor.ts:959-963`'s fallback in `context-safety-middleware.ts:recordUsage`.
3. **Trace data lost (HIGH).** In-memory traces array means OTel + Langfuse + Stdout sinks see nothing for any Vercel-AI-SDK pod. Operator dashboards that aggregate across pods would silently exclude these. Fix: accept a `traceSinks: TraceSink[]` opt; pipe `traces()` array into `sink.emit(entry)` per entry on run completion (or stream emits in real-time).
4. **No env-reader (MEDIUM).** Consumer rebuilds `parseEnv` plumbing — 50+ LOC of duplication that drifts from production. Fix: export a `bootFromEnv(env: NodeJS.ProcessEnv): RunVercelAiAgentTaskInput` helper that re-uses `@kagent/agent-pod`'s `parseEnv`.
5. **No JWT verification (MEDIUM).** Consumer must remember to call `loadCapabilityOptional({ env: process.env })` before constructing the runner input. Forgetting this means accepting unverified JWT claims. Fix: either invoke it inside the runner OR doc-comment + lint-rule the requirement.
6. **No status writer (MEDIUM).** `kubectl get agenttask` would show the AgentTask in `Pending`/`Unknown` because no controller-side patch fires. Substrate's GitOps observability story breaks for Vercel-AI-SDK pods.
7. **Detector flag silently misfires for non-spawn-admitting agents (LOW).** `spawnToolAdmitted` defaults to `true` if undefined; production agent-pod runner derives it from spec. Fix: derive from `admittedToolNames.includes('spawn_child_task')` when not explicitly set.
8. **Permissive `inputSchema: z.unknown()` (LOW).** Tool-call validation is delayed to the inner factory; observed model behavior degrades. Fix: per-tool schema mapping (the report's own `§5 Deviation 3` flagged this as future work).

## 5. Net verdict on "any framework runs in a pod" claim

**Worse than rev2. The claim is now TRUE-WITH-FIVE-CAVEATS.**

The W4 report's claim at `evidence/audit-rev2/W4-Vercel-Ai-REPORT.md:142-146` — "kagent's substrate ships per-pod isolation, A2A messaging, observability, and (as of v0.1.9) a context-window safety-net. Any framework can run in a kagent pod. If you want the safety-net + capability gating + the `context_pressure_ignored` diagnostic, EITHER use `@kagent/agent-loop` OR `@kagent/agent-loop-vercel-ai`" — is **not supported by the adapter as committed**. The adapter would not run a real multi-step agent (Component 5 defect), would not emit observability traces (Component 4 stub), and silently no-ops the safety-net when the gateway omits usage (Component 1 fallback gap). Each issue is fixable; none is fixed today.

The honest framing for `docs/DROP-IN.md` §8:

> kagent's substrate ships per-pod isolation, A2A messaging, observability, and (as of v0.1.9) a context-window safety-net. Any framework can run in a kagent pod. The reference Vercel AI SDK adapter at `packages/agent-loop-vercel-ai` demonstrates the shape of the contract but is NOT production-ready as of v0.1.9.4 — it has a single-step loop defect, no real `TraceSink` wiring, and no `estimateTokens` fallback for usage-less gateways. Use `@kagent/agent-loop` (the reference loop) for the only fully-faithful in-pod runtime today; the Vercel AI SDK adapter is a structural proof, not a drop-in.

**Three concrete recommendations to actually close the rev2 regression** (the rev2 R3.5 recommendations 1-3 still apply, plus):

1. **Fix the `runner.ts:181-193` single-step defect FIRST.** One-line change. After this the adapter at least RUNS multi-step agents.
2. **Wire `traceSinks` into the runner.** Reuse `@kagent/agent-loop`'s `TraceSink` interface; require at least one sink in the input. Without this, observability is silently dropped.
3. **Add a `bootFromEnv` helper that wraps `parseEnv` + `loadCapabilityOptional` + `defineSpawnChildTask` + the runner.** A single function call should produce a runnable Vercel-AI-SDK pod from the same env shape `runAgentTask` consumes — anything less is asking each consumer to rebuild 200+ LOC of substrate plumbing.
4. **Add an integration test that drives the REAL `streamText` against a stub provider** (a `LanguageModelV3` whose `doStream` returns a `ReadableStream` carrying canned `text-delta` and `finish` chunks). This catches the single-step defect, validates the middleware-streamText interaction, and exercises end-to-end the `wrapLanguageModel + onStepFinish + budget extraction + detector` chain. The current `_streamText`-stub tests bypass too much.

## 6. Top 5 surprises (rev3)

1. **The `maxSteps` defect was reachable via a single-line read.** The component 5 review surfaced it within the first read of `runner.ts`. The W4 report's confident "PASS" + "36 tests passing" gave no signal of the integration gap because no test drives the integrated path. This is the cleanest example I've seen of "compiles + tests pass + per-component checklist met but doesn't deliver substrate behavior" — the exact Potemkin pattern the audit prompt asked me to test for.
2. **The runner's own test file admits, in plain English, that its detector test does NOT test the detector.** `runner.test.ts:182-187` — "the test asserts the NEGATIVE shape: the stub path produces a clean run; the detector only fires under realistic middleware accounting." The test name says one thing; the test body says the opposite. This is honest in comments but dishonest in test inventory: the W4 report counts this as a passing detector test.
3. **The trace bridge is genuinely an in-memory stub.** I expected it would be wired to `OtelTraceSink` / `LangfuseTraceSink` from `@kagent/agent-loop`. It is not — `traces()` returns an array, and the runner returns it on the result. Production observability is silently dropped for any Vercel-AI-SDK pod using this adapter today.
4. **The cap wrapper does not verify JWTs and no test caught it.** `wrapToolWithCapabilityCheck` accepts a `CapabilityBundle` directly. JWKS verification is "the caller's job." Production `main.ts:177-209` wires `loadCapabilityOptional` automatically. A consumer who reads only the adapter README and forgets the verify step trusts an unverified JWT.
5. **Component 3's test is the only honest one in the suite.** `run-budget-extractor.test.ts:54-138` runs the REAL `computeQualityFlags` against the REAL extractor output. It would catch a regression. The other five test files mock around exactly the integration boundaries that matter.
