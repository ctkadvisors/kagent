### H6 — safeRestart informer restart (a219b75) — CLOSED-WITH-REGRESSION

**Status: CLOSED-WITH-REGRESSION. Confidence: 90.**

The `createRestarter` at `packages/operator/src/informer-restart.ts:134-195` has a `reset()` method that zeroes `state.attempts`, `state.capReached`, and `pending`. On successful `start()` resolution (`informer-restart.ts:181-184`), only `pending = false` is set. **`state.attempts` is never reset.**

`reset()` is returned from `createRestarter` (`informer-restart.ts:194`) but is never called in `watch.ts` or `job-watch.ts`. The W2-Operator report notes "production callers in watch.ts and job-watch.ts both define `onCapReached` non-conditionally" — correct, but the `reset()` call on successful restart is absent from both callers.

**Consequence:** What the JSDoc calls `maxConsecutiveFailures` is actually `maxTotalNonConcurrentErrors`. An operator with 12 separate transient flaps (each recovered) spread over weeks will permanently wedge the cap and stop all further restart attempts.

**Verification:** `grep -n 'reset()' packages/operator/src/watch.ts` returns no hits. `grep -n 'reset()' packages/operator/src/job-watch.ts` returns no hits. `grep -n '\.reset\(\)' packages/operator/src/` returns only one hit in `task-admission.test.ts:287` (unrelated).

**At-cap behavior:**
- When the cap is reached, `onCapReached?.(err, state.attempts)` fires, which in production (`watch.ts:146-155`) calls `handler.onError?(err)` and `console.error`. The `informerHealth.recordError()` call DOES route through `handler.onError?.(err)` at `watch.ts:150`.
- The healthz `errorsTotal` counter increments on cap reached. However, the `/healthz 200 ok` still fires as long as `msSinceLastEvent()` is within the freshness window — even when the cap is reached and no more restarts are scheduled. A permanently broken informer that had its last event before the cap fired will show `/healthz 503 stale` after 5 minutes, but during those 5 minutes it looks healthy while accepting no new events.

**Task processing during wedged state:** Zero. When the cap is reached, no more `safeRestart` calls are scheduled. AgentTasks sit indefinitely. The operator is effectively dead while appearing alive (for up to 5 minutes on `/healthz`).

**Fix:** In `watch.ts` and `job-watch.ts`, wire `restarter.reset()` in the `informer.on('add', ...)` / `informer.on('update', ...)` handlers (or via a dedicated success callback).

## 2. New HIGH Issues Introduced by Fixes

### NEW-H1 — safeRestart `attempts` counter never resets; `maxConsecutiveFailures` semantic mismatch

**Confidence: 90**

**File:line:** `packages/operator/src/informer-restart.ts:181-184` (success path sets only `pending = false`); `:153-158` (`reset()` exists but is never called in production).

**Evidence:** `reset()` is exported via `createRestarter`'s return value at line 194 but is captured in neither `watch.ts` nor `job-watch.ts`. `grep -rn 'reset()' packages/operator/src/watch.ts` and `job-watch.ts` confirm zero calls.

**Consequence:** `maxConsecutiveFailures=12` is actually "max total independent non-concurrent error events since process start." Twelve independent transient flaps that all recovered trigger permanent cap. The operator loses visibility into all AgentTask state changes. The healthz `/healthz` endpoint reports 200 for up to 5 minutes after the wedge.

**Fix:** In `watch.ts` and `job-watch.ts`, capture the restarter and call `restarter.reset()` from each `informer.on('add', ...)` handler, or pipe `reset` through `InformerRestartLogger.onStartSucceeded` (a new callback).

## 3. Sibling Code Paths with the Same Bug

### SIBLING-1 — `fetchAgent` in supervision-router still does uncached direct API call

**Confidence: 82**

**File:line:** `supervision-router.ts:498-521` (`fetchAgent`).

M2 fixed `fetchParentTask` (lines 450-496) and `fetchTaskByUid` (lines 599-629). `fetchAgent` still calls `customApi.getNamespacedCustomObject` directly on every supervision reconcile. No Agent informer cache exists in the supervision router.

This is less severe than the pre-M2 `fetchParentTask` (a single GET vs. a namespaced LIST), but it is the same category of problem: a cold-path direct API call that bypasses the operator's in-memory cache. In a cluster where many agents fail simultaneously, the supervision router issues one `getNamespacedCustomObject` per failing task.

**Fix:** Add an optional `getAgentByName?: (namespace: string, name: string) => Agent | undefined` dep to `SupervisionRouterDeps`, backed by an Agent informer cache.

### SIBLING-2 — `listSiblings` returns empty when `listChildrenForParent` is not wired

**Confidence: 80**

**File:line:** `supervision-router.ts:523-530`.

`listSiblings` returns `[]` when `deps.listChildrenForParent` is undefined. Production main.ts always wires `listChildrenForParent` (confirmed at `main.ts:1979`), so this is not a WBD instance — but tests that omit `listChildrenForParent` from deps silently drive the supervision engine with zero siblings, which may pass for the wrong reason.

## 5. Operator Config Model Coherence

The rev3 brief asks whether `agentPod.blackboard.acknowledgeUnsafe`, `agentPod.contextSafetyThreshold`, and `supervision.maxEscalationDepth` are consistently shaped.

**Consistency audit:**

| Value | Path in values.yaml | Trapdoor pattern | Helm validation |
|---|---|---|---|
| `contextSafetyThreshold` | `agentPod.contextSafetyThreshold` | Float range (0,1]; OOR → chart fail | `_helpers.tpl:93-101` |
| `contextPressureThreshold` | `agentPod.contextPressureThreshold` | Float range (0,1); OOR → chart fail | `_helpers.tpl:102-109` |
| `blackboard.failOpen` | `agentPod.blackboard.failOpen` | Bool; requires `acknowledgeUnsafe=true` | `_helpers.tpl:129-134` |
| `blackboard.acknowledgeUnsafe` | `agentPod.blackboard.acknowledgeUnsafe` | Bool; paired with `failOpen` | Same block |
| `maxEscalationDepth` | `supervision.maxEscalationDepth` | Int; bad values → operator warn + fallback | None at chart level |

**Finding:** The shapes are NOT uniform. Context thresholds and blackboard fail-open use Helm-level fail-fast guards. The `maxEscalationDepth` has no Helm-level validation — bad values only produce a warn-log at operator runtime. An operator who sets `supervision.maxEscalationDepth: 0` (intending "no limit") silently gets the default 8 instead of a chart-render error. **Confidence 82.**

**Fix:** Add a `hasKey` + range check to `_helpers.tpl:validateValues` for `maxEscalationDepth`.
