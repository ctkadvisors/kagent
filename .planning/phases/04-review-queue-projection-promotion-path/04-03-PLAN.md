  409/422 ladder) and `gateway.ts` (PATCH merge-patch annotation write).
  Five distinct atomic-ish steps per CONTEXT.md D-03 + RESEARCH.md Q3:

    1. Cache-lookup → 404 if missing.
    2. Conflict check → 409 if `review-decision` already set OR if
       re-classifier returns undefined (task not in queue anymore).
    3. (For reason='candidate-template' ONLY): parse + validate YAML →
       `customApi.createNamespacedCustomObject(...)` for the new
       AgentTemplate CR. K8s 409 → 422 with K8s body. K8s 422 → 422
       with K8s body. K8s 5xx → 500 scrubbed.
    4. PATCH AgentTask annotations via `customApi.patchNamespacedCustomObject`
       with `MERGE_PATCH_OPTIONS`. Order matters: CR creation FIRST,
       annotation patch SECOND (D-03 atomicity note).
    5. Emit audit events: `review.accepted` ALWAYS;
       `template.candidate.promoted` ADDITIONALLY on candidate-template
       success.

  The reject handler is simpler: skip step 3; PATCH `review-decision: rejected`;
  emit `review.rejected`.

  The request handler is simplest: skip steps 2/3; PATCH
  `review-requested: "true"` + companion annotations; emit `review.requested`.

  D-02 / D-03 / D-06 enforcement:
    - All 3 POST endpoints fail-closed when `customApi === undefined` (503
      with the verbatim `tasks.ts:147` message). Mirrors the
      `actions.create=false` Helm flag posture.
    - `X-Forwarded-User` is the reviewer-id source; auth middleware in
      main.ts already enforces `WORKBENCH_AUTH_REQUIRED=true` per default.
      Body-supplied `reviewerId` overrides the header per CONTEXT.md
      D-03-A; both fall back to `'unknown'` for annotation/audit.
    - REV-02 single-reviewer scope: NO quorum, NO signed proposals, NO
      no-self-review check. Multi-reviewer is future research per
      REQUIREMENTS.md §4 (CoalitionProposal).
    - D-06 enforcement: agents NEVER write `review-requested` directly.
      The `request` endpoint requires the operator-only auth posture
      (X-Forwarded-User from a Workbench operator session, not from an
      agent JWT).

  Helper-lifting (LM-3): export `extractK8sStatus` and `readCreatedMeta`
  from `tasks.ts` so the accept handler can re-use them. Two consumers
  do NOT justify a shared `routes/k8s-helpers.ts` extract; named exports
