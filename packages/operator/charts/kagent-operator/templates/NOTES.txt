{{- /*
WS-A — security-baseline deprecation warnings. Plaintext values in
agentPod.litellmApiKey.value / agentPod.otlpHeaders.value still WORK
(the deployment renders the plaintext env), but the secret-ref form
is strongly preferred so the operator's PodSpec never carries the
plaintext.
*/}}
{{- if .Values.agentPod.litellmApiKey.value }}

WARNING: agentPod.litellmApiKey.value (plaintext) is set. This renders
the LiteLLM API key directly into the operator's PodSpec, where it
shows up in `kubectl get deploy -o yaml` and in any GitOps audit
trail. Migrate to the secret-ref form:

  agentPod:
    litellmApiKey:
      secretName: <existing-secret-in-{{ .Release.Namespace }}>
      secretKey:  <key-within-that-secret>

(Then `kubectl create secret generic` / Sealed-Secret as appropriate.)
{{- end }}
{{- if .Values.agentPod.otlpHeaders.value }}

WARNING: agentPod.otlpHeaders.value (plaintext) is set — same caveat
as litellmApiKey above. Migrate to:

  agentPod:
    otlpHeaders:
      secretName: <existing-secret>
      secretKey:  <key>
{{- end }}

To inspect the operator's effective env (and confirm the secret-ref
plumbing took effect):

  kubectl --namespace {{ .Release.Namespace }} \
    get deploy {{ include "kagent-operator.fullname" . }} \
    -o jsonpath='{.spec.template.spec.containers[0].env}' | jq
